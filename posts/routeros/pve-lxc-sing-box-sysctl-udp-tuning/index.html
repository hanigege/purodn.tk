PVE LXC sing-box 网关 sysctl 与 UDP 性能优化

整理 PVE 宿主机和 LXC 容器里运行 sing-box 透明代理时的 Linux sysctl 调优思路，重点处理 BBR、UDP 缓冲、队列、连接追踪和 GSO/GRO 检查。

这篇整理 PVE 宿主机或 LXC 容器里跑 sing-box 旁路由、透明代理、网关时常用的 Linux 网络参数。重点不是“把所有看起来能加速的参数都塞进 /etc/sysctl.conf”，而是把确实有意义的部分收敛下来：

  • TCP 出站使用 BBR 和 fq
  • 转发场景打开 IPv4 / IPv6 forwarding
  • 提高监听队列、网卡 backlog、TCP / UDP 缓冲上限
  • 针对 Hysteria2、TUIC、QUIC 这类 UDP 流量检查丢包和 offload
  • 避开已经过时、无效或容易造成副作用的旧 ShadowSocks 时代参数

如果是在 PVE 里跑非特权 LXC，参数要分层处理。宿主机负责桥接、队列、物理网卡、PVE 防火墙路径和全局转发环境；LXC 里运行的 sing-box 才是真正发起代理 TCP / UDP 出站连接的一方，因此容器内也要保留一份能在当前命名空间生效的性能参数。不要只调 PVE 宿主机，也不要把宿主机完整 sysctl 生硬复制到容器里。

1. 先确认环境

先看内核、拥塞控制和当前队列算法：

uname -r
sysctl net.ipv4.tcp_available_congestion_control
sysctl net.ipv4.tcp_congestion_control
sysctl net.core.default_qdisc

如果可用算法里有 bbr，就不要再继续使用 hybla。hybla 是早期面向高延迟链路的算法，放在现代代理网关上通常不是首选。sing-box 作为网关时，TCP 出站、规则下载、系统更新、普通 HTTPS 连接都会受拥塞控制影响，BBR 更适合现在的公网线路。

再确认转发开关：

sysctl net.ipv4.ip_forward
sysctl net.ipv6.conf.all.forwarding
sysctl net.ipv6.conf.default.forwarding

如果 sing-box 只是普通客户端，不承担网关、TProxy、旁路由转发，forwarding 可以不用开。只要它要接管局域网设备流量，就应该打开。

2. 不建议照抄的旧参数

很多网络优化模板里还会出现这些写法：

net.ipv4.tcp_congestion_control = hybla
fs.file-max = 100000
net.ipv4.tcp_tw_reuse = 1
net.ipv4.tcp_max_tw_buckets = 5000

这里要分开看。

tcp_congestion_control = hybla 建议直接改成 bbr。除非你非常明确自己是在卫星链路、极端 RTT 场景里做专项测试，否则没有必要把它作为默认值。

fs.file-max = 100000 也不建议为了“优化”而写死。现代 Linux 的默认值通常已经很大，手动写 10 万反而可能把系统级文件句柄上限压低。真正需要高并发时，应该同时检查服务进程的 systemd LimitNOFILE、shell ulimit -n 和程序本身的连接模型。

tcp_tw_reuse 不再是万能加速开关。新内核对它的语义已经收窄，默认更偏向 loopback 场景；透明代理或转发流量也不靠它解决 TIME_WAIT。个人网关里不写它通常更干净。

tcp_max_tw_buckets = 5000 数值过小。高并发时把 TIME_WAIT 桶压得太低，可能让内核更激进地回收连接状态。除非你已经用 ss -ant state time-wait 和内核日志确认 TIME_WAIT 真的失控，否则不要用很小的值硬压。

3. 推荐 sysctl 配置

在 PVE 宿主机新建独立配置文件，别直接把所有内容塞进 /etc/sysctl.conf，后续回滚和追踪会清楚很多：

nvim /etc/sysctl.d/99-sing-box-gateway.conf

写入下面这份配置：

# ============================================================
# sing-box gateway tuning for PVE / LXC
# 适合 PVE 宿主机承载 sing-box LXC、VM 或旁路由网关的场景。
# ============================================================

# ------------------------------------------------------------
# 1. 路由转发
# sing-box 做旁路由、透明代理、TProxy 网关时需要打开。
# 如果本机只作为普通客户端，不负责转发局域网流量，可以不启用。
# ------------------------------------------------------------
net.ipv4.ip_forward = 1
net.ipv6.conf.all.forwarding = 1
net.ipv6.conf.default.forwarding = 1

# ------------------------------------------------------------
# 2. 队列与 TCP 拥塞控制
# fq + bbr 是现代 Linux 代理网关上更稳妥的默认组合。
# BBR 主要影响 TCP；Hysteria2/TUIC/QUIC 的主链路是 UDP，不要误以为 BBR 会直接优化 UDP 协议本身。
# ------------------------------------------------------------
net.core.default_qdisc = fq
net.ipv4.tcp_congestion_control = bbr

# ------------------------------------------------------------
# 3. 入站队列与监听队列
# netdev_max_backlog 用于网卡收包积压队列，高吞吐或突发流量下可以适当提高。
# somaxconn 和 tcp_max_syn_backlog 影响 TCP 服务端监听队列。
# ------------------------------------------------------------
net.core.netdev_max_backlog = 250000
net.core.somaxconn = 8192
net.ipv4.tcp_max_syn_backlog = 8192

# ------------------------------------------------------------
# 4. TCP / UDP 缓冲上限
# rmem_max / wmem_max 是 socket 接收/发送缓冲的系统上限。
# QUIC、Hysteria2、TUIC 走 UDP，高吞吐时最容易先撞到 UDP 接收缓冲不足。
# ------------------------------------------------------------
net.core.rmem_max = 67108864
net.core.wmem_max = 67108864
net.core.rmem_default = 262144
net.core.wmem_default = 262144

# ------------------------------------------------------------
# 5. TCP 自动调优范围
# 三个值分别是最小值、默认值、最大值。
# 这里保留较大的上限，允许高延迟大带宽链路自动扩大窗口。
# ------------------------------------------------------------
net.ipv4.tcp_rmem = 4096 87380 67108864
net.ipv4.tcp_wmem = 4096 65536 67108864

# ------------------------------------------------------------
# 6. UDP 最小缓冲
# udp_rmem_min 对 UDP 收包有意义，能降低高吞吐 UDP 被小缓冲拖住的概率。
# udp_wmem_min 在新内核文档里已标注发送端不再使用，不作为推荐项写入。
# ------------------------------------------------------------
net.ipv4.udp_rmem_min = 16384

# ------------------------------------------------------------
# 7. 基础连接健壮性
# syncookies 用于 SYN flood 防护。
# tcp_mtu_probing 可以缓解部分路径 MTU 黑洞导致的网页卡住或 TLS 握手异常。
# tcp_fastopen 对支持 TFO 的 TCP 连接有帮助，不影响 UDP 主链路。
# ------------------------------------------------------------
net.ipv4.tcp_syncookies = 1
net.ipv4.tcp_mtu_probing = 1
net.ipv4.tcp_fastopen = 3

# ------------------------------------------------------------
# 8. 本地主动连接端口范围
# 网关、代理、下载器同机运行时，本地临时端口范围可以放宽。
# ------------------------------------------------------------
net.ipv4.ip_local_port_range = 10000 65000

# ------------------------------------------------------------
# 9. TCP 连接保活与关闭等待
# 只做温和收敛，不使用过激 TIME_WAIT 回收参数。
# ------------------------------------------------------------
net.ipv4.tcp_fin_timeout = 30
net.ipv4.tcp_keepalive_time = 1200

应用配置：

sysctl --system

如果只想应用这一个文件：

sysctl -p /etc/sysctl.d/99-sing-box-gateway.conf

验证关键项：

sysctl net.ipv4.tcp_congestion_control
sysctl net.core.default_qdisc
sysctl net.core.rmem_max net.core.wmem_max
sysctl net.ipv4.udp_rmem_min

预期至少能看到：

net.ipv4.tcp_congestion_control = bbr
net.core.default_qdisc = fq

4. LXC 里怎么处理

非特权 LXC 容器里不要强行把宿主机完整 sysctl 写一遍。很多参数会报错，例如：

Read-only file system
permission denied on key

正确思路是：

  1. 宿主机写 99-sing-box-gateway.conf
  2. 宿主机额外写旁路网关专用的 98-sing-box-pve-bypass.conf
  3. 宿主机 sysctl --system
  4. LXC 内只写能在容器命名空间生效的性能参数
  5. sing-box 配置、路由规则、nftables / iptables、OpenRC 或 systemd 服务限制仍然在容器内维护

容器内可以执行：

sysctl net.ipv4.ip_forward
sysctl net.ipv4.tcp_congestion_control
sysctl net.core.rmem_max net.core.wmem_max

如果 LXC 作为独立网关命名空间，ip_forward 在容器内也要确认是 1。但 net.core.* 这类底层缓冲和队列参数，通常应以宿主机为准。

生产环境里更稳的做法是给 LXC 单独写一个性能文件，不和 TProxy 转发文件混在一起：

nvim /etc/sysctl.d/98-sing-box-performance.conf

写入：

# sing-box 网关性能参数。容器或 VM 内的 sing-box 才是真正发起代理 TCP/UDP 出站连接的一方，不能只依赖宿主机 BBR。
net.ipv4.tcp_congestion_control = bbr
net.ipv4.tcp_rmem = 4096 131072 67108864
net.ipv4.tcp_wmem = 4096 65536 67108864
net.ipv4.udp_rmem_min = 16384
net.ipv4.tcp_slow_start_after_idle = 0

应用并检查：

sysctl -p /etc/sysctl.d/98-sing-box-performance.conf
sysctl net.ipv4.tcp_available_congestion_control
sysctl net.ipv4.tcp_congestion_control
sysctl net.ipv4.tcp_rmem net.ipv4.tcp_wmem
sysctl net.ipv4.udp_rmem_min

如果 tcp_available_congestion_control 里没有 bbr，就不要硬写 tcp_congestion_control = bbr。可以先注释这一行，只保留缓冲和 tcp_slow_start_after_idle：

# net.ipv4.tcp_congestion_control = bbr

TProxy 自身的转发参数建议继续放在另一份文件里，例如 /etc/sysctl.d/99-sing-box-tproxy.conf：

net.ipv4.ip_forward=1
net.ipv4.conf.all.rp_filter=0
net.ipv4.conf.default.rp_filter=0
net.ipv4.conf.eth0.rp_filter=0
net.ipv6.conf.all.forwarding=1
net.ipv6.conf.default.forwarding=1
net.ipv6.conf.eth0.forwarding=1
net.ipv6.conf.eth0.accept_ra=2
net.ipv6.conf.eth0.accept_ra_defrtr=1

这样分开以后，UI 或脚本重写 TProxy 参数时，不会覆盖性能参数；后续回滚时也能判断到底是转发问题还是吞吐问题。

5. PVE 旁路网关专用补丁

PVE 自带 firewall 和默认 sysctl 可能会把 rp_filter 设成 2。普通虚拟机这样通常没问题，但旁路网关、TProxy、策略路由、FakeIP 回环这类场景可能出现非对称路径。为了避免 PVE 桥接层误丢包，建议给承载 sing-box 的宿主机加一份更早加载的覆盖文件：

nvim /etc/sysctl.d/98-sing-box-pve-bypass.conf

写入：

# sing-box 旁路网关的 PVE 宿主参数。透明代理/策略路由可能出现非对称路径，rp_filter 必须关闭，避免 PVE 桥接层误丢包。
net.ipv4.conf.all.rp_filter = 0
net.ipv4.conf.default.rp_filter = 0
net.ipv4.conf.vmbr0.rp_filter = 0
net.ipv4.conf.veth116i0.rp_filter = 0
net.ipv4.conf.fwbr116i0.rp_filter = 0
net.ipv4.conf.fwpr116p0.rp_filter = 0
net.ipv4.conf.fwln116i0.rp_filter = 0

这里的 veth116i0、fwbr116i0、fwpr116p0、fwln116i0 要按自己的 CTID 修改。比如容器 ID 是 101，接口通常会变成 veth101i0。如果这个 LXC 已经关闭 PVE firewall，运行态可能只剩 veth116i0，没有 fwbr/fwpr/fwln，文件里保留这些项也不会影响已存在接口；但执行 sysctl -p 时不存在的接口会报错。生产里更稳的方式是按当前实际接口写，或者只写必定存在的 all/default/vmbr0/vethXXXi0。

应用并检查：

sysctl -p /etc/sysctl.d/98-sing-box-pve-bypass.conf
sysctl net.ipv4.conf.all.rp_filter
sysctl net.ipv4.conf.default.rp_filter
sysctl net.ipv4.conf.vmbr0.rp_filter
sysctl net.ipv4.conf.veth116i0.rp_filter

如果这台 LXC 专门做 sing-box 网关，且不依赖 PVE firewall 规则过滤入站，可以考虑关闭该 LXC 的 PVE firewall，减少流量经过 fwbr/fwpr/fwln 这一层：

pct set 116 -net0 name=eth0,bridge=vmbr0,firewall=0,gw=10.20.20.1,gw6=fd88::1111,hwaddr=BC:24:11:1E:91:3C,ip=10.20.20.6/24,ip6=fd88::6666/64,type=veth

这条命令里的 IP、网关、MAC、IPv6 都必须按现有 pct config 116 原样改写，不能照抄。改完确认：

pct config 116 | grep net0
ip -br link | grep -E 'veth116|fwbr116|fwpr116|fwln116'

理想状态是 net0 里显示 firewall=0，运行态只剩 veth116i0。如果还能看到 fwbr116i0、fwpr116p0、fwln116i0，说明这条链路还在 PVE firewall 路径里。

6. UDP 丢包检查

Hysteria2、TUIC、QUIC 不像传统 TCP 那样由内核 TCP 栈负责拥塞控制。系统层最常见的问题是 UDP socket buffer 不够，内核直接丢包。

运行：

ss -unmp

重点看 sing-box 或相关进程对应的 UDP socket 有没有 drops 增长。也可以配合连续观察：

watch -n 1 'ss -unmp | grep -E "sing-box|drops|udp"'

如果测速或高峰下载时 drops 明显增加，优先检查：

  • net.core.rmem_max
  • net.core.rmem_default
  • net.ipv4.udp_rmem_min
  • sing-box / Hysteria2 / TUIC 自身是否有 socket buffer 配置项
  • PVE 宿主机网卡队列、虚拟网卡 offload、CPU 占用

不要只盯着 BBR。BBR 对 UDP 主链路没有直接作用。

7. GSO / GRO / UDP segmentation 检查

Hysteria2 和 TUIC 在高吞吐场景里会受 UDP GSO 影响。GSO 能减少用户态和内核之间处理大量小 UDP 包的开销，CPU 占用会低一些。

先在 PVE 宿主机确认网卡名：

ip link

物理网卡可能是 eno1、enp3s0，LXC 对应的虚拟网卡可能类似 veth100i0。查看 offload：

ethtool -k eno1
ethtool -k veth100i0

重点看这些项：

generic-segmentation-offload
generic-receive-offload
tx-udp-segmentation

如果内核和驱动支持，可以尝试开启：

ethtool -K eno1 gso on gro on
ethtool -K eno1 tx-udp-segmentation on
ethtool -K veth100i0 gso on gro on
ethtool -K veth100i0 tx-udp-segmentation on

有些网卡、veth 或驱动不支持 tx-udp-segmentation，会返回不支持。这不是 sysctl 能解决的问题。先记录实际能力，再看是否需要换内核、调整虚拟化网络模型，或者降低单连接 UDP 吞吐预期。

如果要开机自动设置，可以放到 PVE 宿主机的 systemd oneshot 服务里。不要写到 LXC 容器里，因为容器通常管不到宿主机的物理网卡和 veth offload。

8. systemd / OpenRC 文件句柄限制

sysctl 只解决内核网络栈的一部分。高并发代理还要看 sing-box 服务的文件描述符限制：

systemctl show sing-box -p LimitNOFILE

如果数值太小，可以给服务加 override：

systemctl edit sing-box

写入：

[Service]
LimitNOFILE=1048576

应用：

systemctl daemon-reload
systemctl restart sing-box
systemctl show sing-box -p LimitNOFILE

这比把 fs.file-max 固定成 100000 更准确。系统级上限、服务级上限和程序实际打开文件数是三件事，不要混在一起调。

Alpine/OpenRC 环境没有 systemd override，要看 init 脚本里是否设置了 ulimit -n，以及 LXC 容器本身允许的 hard limit：

rc-service sing-box status
cat /etc/init.d/sing-box
cat /proc/$(pidof sing-box | awk '{print $1}')/limits

如果容器的 hard limit 只有 1048576，OpenRC 里就不要再写更高的值。服务级限制应该匹配容器上限，避免启动脚本看起来很大、实际进程拿不到。

9. 排查顺序

如果优化后速度还是不稳，按这个顺序查：

  1. sing-box check 确认配置无误
  2. systemctl status sing-box 或 rc-service sing-box status 看服务是否重启失败
  3. ss -unmp 看 UDP socket drops
  4. top / htop 看 CPU 是否打满单核
  5. ethtool -k 看 GSO / GRO / tx-udp-segmentation
  6. sysctl net.ipv4.tcp_congestion_control net.core.default_qdisc 确认 BBR / fq 生效
  7. 云厂商安全组、PVE 防火墙、宿主机防火墙确认 UDP 端口放行

如果是 Hysteria2 或 TUIC，服务端和客户端的带宽参数也要合理。把 up_mbps、down_mbps 写得远高于真实线路，不一定更快，反而可能让协议更激进地制造丢包和抖动。

10. 小结

这套配置的核心是“保守、可验证、适合长期运行”：

  • hybla 改成 bbr
  • 不再写死过小的 fs.file-max
  • 不把 tcp_tw_reuse 和 tcp_max_tw_buckets 当成必选项
  • 对 UDP 增加接收缓冲和系统缓冲上限
  • 把 PVE 宿主机、LXC 容器、PVE firewall 路径、虚拟网卡 offload 分开处理

真正的 UDP 性能优化不只是 sysctl。ss -unmp 看到的 drops、ethtool -k 看到的 offload 能力、PVE 宿主机 CPU 和网卡队列，往往比单纯堆参数更关键。
