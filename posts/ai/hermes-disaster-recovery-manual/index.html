Hermes 灾难恢复手册 — 从零手动重建所有服务

如果所有 Hermes bot 全部挂了，怎么从在线仓库一步步手动恢复。小白友好，复制粘贴就能用。

如果你的 Hermes bot 全部失联，网页端也连不上，别慌。这份教程教你从头手动恢复，每一步都有完整的命令，复制粘贴就能跑。

适用场景

  • 所有 Hermes bot 都不回复消息
  • SSH 还能连上服务器
  • GitHub 还能访问（备份仓库在线）

前提

  • 服务器 IP：10.20.20.19
  • 登录用户：hermesuser
  • 备份仓库：git@github.com:hanigege/hermes.git

第一步：SSH 登录服务器

在你的电脑上打开终端（Windows 用 PowerShell 或 CMD），输入：

ssh hermesuser@10.20.20.19

登录成功后会看到类似这样的提示：

hermesuser@hermes:~$

第二步：确认系统状态

先看看系统跑了多久、磁盘空间够不够：

uptime
df -h

磁盘应该还有 至少 10G 空闲。如果空间不够，先清理一下。

第三步：从 GitHub 克隆备份仓库并一键恢复

📦 仓库里有什么：

  • hermes-agent/ 全部源码（4000+ 文件）✅
  • config.yaml + .env 配置文件 ✅
  • systemd-templates/ 优化版服务模板 ✅
  • scripts/recover.sh 一键恢复脚本 ✅
  • ❌ venv/（需要重建）
  • ❌ sessions/（聊天历史，体积大，不备份）

把整个 Hermes 配置和代码从 GitHub 拉下来，然后跑恢复脚本——仅需两步：

# 第一步：克隆仓库
cd ~
git clone git@github.com:hanigege/hermes.git .hermes

# 第二步：一键恢复（重建 venv + 安装服务 + 启动）
bash ~/.hermes/scripts/recover.sh

如果提示 Permission denied (publickey)，说明 SSH key 没配好。可以用 HTTPS 方式：

git clone https://github.com/hanigege/hermes.git .hermes

恢复脚本会自动完成：

  1. 重建 Python 虚拟环境
  2. 复制优化版 systemd 服务文件
  3. 重载 systemd
  4. 启动所有 5 个 Hermes 服务
  5. 显示最终状态

全部显示 active 即恢复成功 🎉

第四步：检查 API key

恢复脚本完成后，确认一下 API key 不是占位符：

grep -c "sk-.*" ~/.hermes/.env ~/.hermes/profiles/*/.env

如果返回 0，说明 key 需要重新填入：

nvim ~/.hermes/.env
nvim ~/.hermes/profiles/code/.env
nvim ~/.hermes/profiles/shaoji/.env
nvim ~/.hermes/profiles/singbox/.env
nvim ~/.hermes/profiles/synapse/.env

需要填入的 key：

  • `DEEPSEEK_API_KEY=*** 从 DeepSeek 官网获取）
  • `TELEGRAM_BOT_TOKEN=*** 从 @BotFather 获取）

填完后重启服务使新 key 生效：

systemctl --user restart hermes-gateway hermes-gateway-code hermes-gateway-shaoji hermes-gateway-singbox hermes-gateway-synapse

第五步：测试机器人是否在线

现在去 Telegram 给你的 bot 发一条消息试试。如果正常回复，说明恢复成功了。

如果还是不回复，检查日志看看是什么问题：

journalctl --user -u hermes-gateway.service --since "5 min ago" --no-pager | grep -i "error\|402\|Insufficient"

常见的日志错误：

日志内容 原因 解决方法
HTTP 402: Insufficient Balance DeepSeek API 余额不足 去 DeepSeek 官网充值
httpx.ConnectError Telegram 网络不通 检查外网连通性，等网络恢复即可
Unable to locate executable Python 环境没装好 重跑 bash ~/.hermes/scripts/recover.sh

优化版 systemd 服务文件

各服务的配置说明，排查问题时可对照查看。

关键优化点

配置项 作用
--profile <name> 指定运行哪个 profile，5 个服务各跑各的
ExecStartPre 启动前检查 venv 完整性 + 清理锁文件，防止空转
RestartSteps=5 + RestartMaxDelaySec=300 指数退避重启（5s→25s→125s→300s），防崩溃风暴
ExecReload 支持 systemctl reload 热重载，不用重启
KillMode=mixed 先发 SIGTERM，超时后 SIGKILL，干净退出
TimeoutStopSec=210 给够 3.5 分钟优雅关闭时间
MemoryMax=2G + MemoryHigh=1G 内存安全阀，防止泄漏拖垮系统
NoNewPrivileges=yes 防提权攻击
PrivateTmp=yes + ProtectSystem=full 沙箱隔离

生产配置通过 systemd drop-in 覆盖层（10-production.conf）实现，这样 Hermes 自身更新主文件时不会冲掉加固参数。

蜜（default）

路径：~/.config/systemd/user/hermes-gateway.service

主文件：

[Unit]
Description=Hermes Agent Gateway - Messaging Platform Integration
After=network-online.target
Wants=network-online.target
StartLimitIntervalSec=0

[Service]
Type=simple
ExecStart=/home/hermesuser/.hermes/hermes-agent/venv/bin/python -m hermes_cli.main --profile default gateway run --replace
WorkingDirectory=/home/hermesuser/.hermes
Environment="PATH=/home/hermesuser/.hermes/hermes-agent/venv/bin:/usr/bin:/home/hermesuser/.local/bin:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin"
Environment="VIRTUAL_ENV=/home/hermesuser/.hermes/hermes-agent/venv"
Environment="HERMES_HOME=/home/hermesuser/.hermes"
Restart=always
RestartSec=5
RestartMaxDelaySec=300
RestartSteps=5
RestartForceExitStatus=75
KillMode=mixed
KillSignal=SIGTERM
ExecReload=/bin/kill -USR1 $MAINPID
TimeoutStopSec=210
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=default.target

生产覆盖层（~/.config/systemd/user/hermes-gateway.service.d/10-production.conf）：

[Unit]
StartLimitIntervalSec=300
StartLimitBurst=20

[Service]
# 启动前检查
ExecStartPre=/usr/bin/test -x /home/hermesuser/.hermes/hermes-agent/venv/bin/python
ExecStartPre=/usr/bin/rm -f /home/hermesuser/.hermes/gateway.lock /home/hermesuser/.hermes/gateway.pid

# 资源限制（正常跑 ~100-180MB，2G 是安全阀）
MemoryAccounting=yes
CPUAccounting=yes
TasksAccounting=yes
MemoryHigh=1G
MemoryMax=2G
TasksMax=256
LimitNOFILE=65536

# 安全加固
UMask=0077
NoNewPrivileges=yes
PrivateTmp=yes
ProtectSystem=full
RestrictSUIDSGID=yes
LockPersonality=yes

# 日志限流
LogRateLimitIntervalSec=60
LogRateLimitBurst=1000

佬（code）

路径：~/.config/systemd/user/hermes-gateway-code.service

相比 default 的改动：

  • ExecStart 加 --profile code
  • WorkingDirectory 改为 /home/hermesuser/.hermes/profiles/code
  • HERMES_HOME 改为 /home/hermesuser/.hermes/profiles/code

烧鸡（shaoji）

路径：~/.config/systemd/user/hermes-gateway-shaoji.service

相比 default 的改动：

  • ExecStart 加 --profile shaoji
  • HERMES_HOME 改为 /home/hermesuser/.hermes/profiles/shaoji

synapse

路径：~/.config/systemd/user/hermes-gateway-synapse.service

与 shaoji 结构完全相同，把 shaoji 替换为 synapse。

singbox（网络代理）

路径：~/.config/systemd/user/hermes-gateway-singbox.service

与 shaoji 结构完全相同，把 shaoji 替换为 singbox。

所有 5 个服务共享同一套 drop-in 生产覆盖层，所以加固参数（内存限制、安全沙箱、日志限流、ExecStartPre）完全一致。主文件只写基础配置，生产配置由 drop-in 统一管理。

备份仓库内容速查表

仓库里有的 ✅

文件/目录 说明
hermes-agent/ 源码 (4172 文件) 全部 Python 源码 + pyproject.toml + uv.lock
config.yaml 主配置（备份时会自动剥离 api_key）
.env 主配置 + 各 profile .env 环境变量，部分 API key 可能是占位符
skills/ 所有自定义技能
profiles/ 各 profile 的配置、状态、缓存

备份时排除的 ❌

备份脚本 /home/hermesuser/.hermes/scripts/backup-hermes.sh 会排除以下内容：

排除项 原因
hermes-agent/ 源码已完整在 git 中，无需重复备份
venv/ 可重建，体积大
node_modules/ npm 依赖，可重建
__pycache__/ Python 缓存，自动生成
**/.cache/huggingface/ 模型缓存，体积大
*.tmp 临时文件
node/ Node.js 运行时文件
.git/ Git 内部数据

不备份的运行时数据 ❌

数据 原因
sessions/（聊天历史） 体积太大，不备份。恢复后历史会话不可用，但机器人可以正常对话
state.db（状态数据库） 运行时状态，恢复后重新生成
memory_store.db（记忆库） 运行时记忆，恢复后重新生成
logs/ 日志文件，恢复后重新生成

恢复之后： 会话历史和记忆会丢失，但机器人能正常对话，重新聊几次记忆就会慢慢恢复。

注意事项

  1. 不要重启服务器 — 如果某个 bot 没起来，单独重启它的 service 就行，不需要 reboot
  2. API key 可能在 .env 里是占位符 — 克隆仓库后 .env 里的 key 可能是 ***，需要重新填入
  3. .env 文件权限要严格 — 确保别人读不到： 
    chmod 600 ~/.hermes/.env ~/.hermes/profiles/*/.env
  4. systemd 服务文件需要安装时生成 — 不要手动写，用 hermes gateway install 命令
  5. 每个 profile 有自己的 .env — 不要只配了主配置，忘了配 code、shaoji、singbox、synapse 的
