Matrix Synapse media_store 分层清理：保留小文件，自动清大文件

解释 Synapse 媒体仓库为什么会长期增长，并整理一套适合 Docker / Alpine 生产环境的分层清理方案：5MiB 以内文件长期保留，超过 5MiB 的文件 90 天后清理，超过 100MiB 的超大文件 7 天后清理，远程缓存 7 天清理。

自建 Matrix Synapse 跑久之后，/root/data/docker_data/synapse/data/media_store/ 可能比数据库还大。更容易让人困惑的是：明明已经在 Element 聊天窗口里手动删除过图片、视频或消息，服务端磁盘却没有明显变小。

这是正常现象。Matrix 里的消息事件、媒体文件和媒体缩略图不是同一个东西。客户端里删除或撤回消息，通常只是让房间事件变成 redaction 状态；Synapse 的媒体仓库仍然可能保留本地上传文件、缩略图、远程媒体缓存和 URL 预览缓存。要控制磁盘，应该单独管理 media store。

本文整理一套适合个人或小团队 Synapse 的生产策略：

  • 限制以后单个上传大小，避免再出现几百 MB 甚至 1GB 的媒体文件。
  • 5MiB 以内文件长期保留，尽量保护语音、小图和普通小文件。
  • 超过 5MiB 的文件 90 天后清理，超过 100MiB 的超大文件 7 天后清理。
  • 远程媒体缓存 7 天后自动清理。
  • 加密房间里的过期大文件被清理后，旧消息点开可能显示不可用；需要提前告知用户清理规则。
  • 孤立文件不做即时硬删；需要做时先报告、再隔离、最后延迟删除。

官方参考：

  • Synapse 配置手册：max_upload_size 与 media_retention
    https://element-hq.github.io/synapse/latest/usage/configuration/config_documentation.html
  • Synapse Media Admin API：删除本地媒体、清理远程缓存
    https://element-hq.github.io/synapse/latest/admin_api/media_admin_api.html

一、先确认到底谁占空间

不要一上来就删数据库。先看目录大小：

du -h -d 2 /root/data/docker_data/synapse 2>/dev/null | sort -h

如果输出类似：

64M   /root/data/docker_data/synapse/postgres
9.6G  /root/data/docker_data/synapse/data/media_store
9.7G  /root/data/docker_data/synapse

说明大头不是聊天数据库，而是媒体仓库。

继续拆 media store：

du -h -d 1 /root/data/docker_data/synapse/data/media_store 2>/dev/null | sort -h

常见目录含义：

目录 含义
local_content/ 本服用户上传的原始媒体，最容易变大
local_thumbnails/ 本服媒体缩略图
remote_content/ 从其他 Matrix homeserver 下载缓存的远程媒体
url_cache/ URL 预览缓存
url_cache_thumbnails/ URL 预览缩略图

再看最大文件：

find /root/data/docker_data/synapse/data/media_store -type f -exec du -h {} + 2>/dev/null | sort -h | tail -50

如果里面出现很多 300M、500M、1.2G 的 local_content/... 文件，通常就是用户在 Matrix 里发过大视频、大压缩包或误把 Matrix 当文件传输用了。

数据库大小可以这样看：

docker exec postgres16 psql -U synapse_user -d synapse -c \
  "SELECT pg_size_pretty(pg_database_size(current_database())) AS synapse_db_size;"

如果数据库只有几十 MB，而 media store 几个 GB，清理重点就是媒体，不是 PostgreSQL。

二、为什么客户端删了服务端还在

Element 里的“删除消息”“撤回消息”和 Synapse 后端的媒体文件删除不是同一件事。

简单理解：

  • 消息事件在数据库里。
  • 媒体文件在 media_store/ 里。
  • 缩略图是 Synapse 另外生成的派生文件。
  • 远程媒体和 URL 预览是缓存。

删除或撤回消息后，旧消息在客户端里看不见了，但媒体仓库未必马上删除对应文件。对于加密房间，服务端也不能像普通相册应用一样理解每个附件的业务含义。Synapse 提供的是单独的媒体保留策略和 Admin API。

所以生产环境要把“消息清理”和“媒体清理”分开处理。

三、retention 不等于删除媒体文件

很多人会在 homeserver.yaml 里配置消息保留策略，例如：

retention:
  enabled: true
  default_policy:
    max_lifetime: 3d
  allowed_lifetime_max: 3d
  purge_jobs:
    - interval: 1d

这段配置处理的是房间事件，也就是聊天历史里的消息记录。它的作用可以理解成：

retention -> 清数据库里的消息事件

但一条带图片、视频或文件的 Matrix 消息通常由两部分组成：

消息事件：
  “这里有一个附件，mxc://666666.xyz/abc123”

媒体文件：
  /data/media_store/local_content/.../abc123

retention 到期后，Synapse 会清理数据库里的消息事件；这不等于立刻删除 media_store/local_content/ 里的原始文件。因此你可能会看到：聊天窗口里的消息已经删了，甚至数据库也不大，但服务端 media_store 仍然有好几个 GB。

要形成闭环，需要两套策略一起用：

retention      -> 清消息事件
media_retention/Admin API -> 清媒体文件

如果你的目标是“90 天后聊天记录不保留，同时避免媒体文件无限增长”，可以这样组合：

retention:
  enabled: true
  default_policy:
    max_lifetime: 90d
  allowed_lifetime_max: 90d
  purge_jobs:
    - interval: 1d

再配合后文的安全脚本：

本地上传媒体：5MiB 以内文件长期保留，超过 5MiB 的文件 90 天后清理，超过 100MiB 的超大文件 7 天后清理
远程媒体缓存：按 7 天清理
URL 预览缓存：按 Synapse 配置或单独维护

这样含义才是：

90 天后清消息事件
每天按分层规则清理本地大媒体和可重新拉取的远程缓存

这里要接受一个取舍：如果房间事件里还写着 mxc://116160.xyz/abc123，而 abc123 的文件已经按规则到期清理，客户端以后打开旧文件时可能遇到 404。Element X Android 对这类 404 有时会显示成视频播放错误。所以规则必须提前告知用户：5MiB 以内文件长期保留，超过 5MiB 的文件 90 天后清理，超过 100MiB 的超大文件 7 天后清理。

四、限制以后单个上传大小

编辑：

vi /root/data/docker_data/synapse/data/homeserver.yaml

增加或调整：

max_upload_size: 20M

个人服务可以设得更小：

max_upload_size: 10M

Synapse 官方默认值是 50M。如果你只用于文字、图片、小文件，10M 或 20M 更安全。

如果前面有 Nginx Proxy Manager、Nginx、Caddy 之类反代，也要让反代的上传限制和 Synapse 一致。以本文 Matrix 入口为例，真正要改的是 NPM UI 里 matrix.666666.xyz 这个 Proxy Host：

NPM 后台
-> Hosts
-> Proxy Hosts
-> 找到 matrix.666666.xyz
-> 点右侧三个点
-> Edit
-> Advanced

在 Advanced 的文本框里添加：

client_max_body_size 20M;

保存后，NPM 会把这段配置持久保存到自己的数据库 advanced_config，再生成到对应的 nginx 配置文件里。这样以后你在 UI 里改证书、开关 WebSocket 或重新保存 Proxy Host，这个上传限制也不会丢。

这里要特别注意：不要把主要步骤写成“手工修改 NPM 生成出来的 nginx 文件”，例如：

/root/data/docker_data/npm/data/nginx/proxy_host/*.conf

proxy_host/*.conf 里面的编号是 NPM 内部生成的，不同机器可能不一样。它只能作为排障时确认最终生成结果的地方，不应该作为教程里的主要配置入口。

如果你明确要允许 2G 大文件，Synapse 和 NPM 要一起放大：

max_upload_size: 2G

然后在 NPM UI 的 matrix.666666.xyz -> Edit -> Advanced 里写：

client_max_body_size 2048m;

改完重启 Synapse：

cd /root/data/docker_data/synapse
docker compose up -d

五、本地媒体用脚本做分层清理

Synapse 的 media_retention 按“最后访问时间”清理媒体。注意是最后访问时间，不是创建时间；如果媒体从来没有被访问过，才会使用创建时间。

它适合清远程缓存，但不适合直接拿来表达本服用户上传附件的分层策略。本文最终使用 Admin API 脚本实现：

<=5MiB：长期保留
>5MiB：90 天后清理
>100MiB：7 天后清理
远程缓存：7 天后清理

media_retention.local_media_lifetime 只能按时间清本地媒体，不能按单个文件大小分层。比如你设置：

media_retention:
  local_media_lifetime: 30d

那么 30 天没访问的小语音、小图片也会一起进入清理范围，这和“5MiB 以内长期保留”的目标冲突。

所以推荐做法是：不要给本地媒体配置统一的 local_media_lifetime，本服用户上传的 local_content/ 交给后文脚本按大小和时间分层清理。

远程缓存可以以后再单独配置，例如：

media_retention:
  remote_media_lifetime: 30d

远程缓存可以按 7 天清；如果加入的远程房间很多，又不想频繁重新拉取图片，可以改成 30 天或 60 天。

如果你修改了 homeserver.yaml，改完重启 Synapse：

cd /root/data/docker_data/synapse
docker compose up -d

六、为什么不能只按大小删除本地媒体

Synapse 的媒体删除接口很方便，但它只知道媒体仓库，不知道客户端会怎样解释旧消息。对本服用户上传的 local_content/，最危险的做法是按大小和时间直接调用：

POST /_synapse/admin/v1/media/delete?before_ts=<before_ts>&size_gt=<size_gt>&keep_profiles=true

这个接口会删除命中的本地媒体文件和对应缩略图。问题在于：房间里的消息事件可能仍然存在，而且消息内容里还引用着同一个 mxc://。对于未加密房间，服务端可以从事件 JSON 里搜索 mxc://server/media_id；对于加密房间，服务端只能看到 m.room.encrypted，不能直接理解附件内容。客户端却能解密消息，所以它以后仍会尝试加载那个 MXC。

结果就是：媒体文件删得很干净，但 Element X Android 打开时间线或视频时会看到 404，并可能弹出“视频播放错误”。这不是删除调度不够精确，而是删除对象不完整：只删了媒体，没有处理引用它的消息。

如果要做到“旧消息也不留下不可用附件”的最好体验，安全删除顺序应该是：

1. 找到或确认引用该 mxc:// 的消息事件。
2. 先撤回消息，或确认 retention 已经清掉对应房间事件。
3. 再删除媒体文件和缩略图。
4. 再删除媒体文件和缩略图。

朋友提到的“上传成功后按 mxcid 记录元数据，生成到期时间，到点再触发删除”适合解决调度问题，误差可以做到很小。但加密房间里服务端无法可靠反查消息引用，所以本文最终选择更容易落地的规则：按大小和时间到期清理，并提前告知用户清理规则。

远程媒体缓存不同。remote_content/ 是从其他 homeserver 拉下来的缓存，通常可以重新获取，更适合按 30 天或 60 天自动清。URL 预览缓存也更接近缓存，可以单独维护。本地上传媒体则应更保守。

七、准备管理员 token

Admin API 需要一个 Synapse 管理员用户的 access token。

临时测试时，可以直接用现有管理员账号的 token。常见获取方式：

  1. 用管理员账号登录 Element。
  2. 打开设置。
  3. 找到帮助与关于。
  4. 展开高级。
  5. 复制 Access Token。

不要把 token 写进公开文章、截图或 shell history。建议放到服务器文件里：

mkdir -p /root/.config/synapse-admin
vi /root/.config/synapse-admin/access_token
chmod 600 /root/.config/synapse-admin/access_token

文件内容只放 token 本身，不要加引号。

如果 Synapse 在宿主机映射了 8008，脚本优先走本机地址：

http://127.0.0.1:8008

这样 token 不需要经过公网和反代。

生产里更推荐给 cron 单独创建一个专用管理员账号，例如：

@media_purge_admin:example.com

然后用这个账号登录一次，把拿到的 token 保存到：

/root/.config/synapse-admin/access_token

原因是实际运行中很容易遇到旧 token 失效。比如管理员账号在 Element 里退出登录、删除设备、修改密码、清理会话，或者迁移/恢复数据库后设备记录变化，都可能让旧 access token 不能再用。脚本跑了一段时间后，Admin API 可能返回：

{"errcode":"M_UNKNOWN_TOKEN","error":"Invalid access token passed.","soft_logout":false}

这时不是清理接口坏了，而是 token 已失效，需要重新登录管理员账号刷新 token。

专用管理员账号的 token 也不是永不过期；它只是更适合 cron 使用：

  • 不会因为你日常在 Element 里退出管理员账号而顺手把 cron token 弄失效。
  • 不会和手机、桌面端、浏览器里的日常登录设备混在一起。
  • 出问题时一眼就知道这个 token 只服务于媒体清理脚本。

检查 token 是否有效

最简单的检查方式是请求 Matrix Client API 的 /account/whoami：

TOKEN_FILE="/root/.config/synapse-admin/access_token"
TOKEN="$(cat "$TOKEN_FILE")"

curl -sS \
  -H "Authorization: Bearer ${TOKEN}" \
  http://127.0.0.1:8008/_matrix/client/v3/account/whoami

正常会返回类似：

{
  "user_id": "@media_purge_admin:example.com",
  "device_id": "MEDIA_PURGE_CRON",
  "is_guest": false
}

如果 token 失效，会返回类似：

{
  "errcode": "M_UNKNOWN_TOKEN",
  "error": "Invalid access token passed.",
  "soft_logout": false
}

还可以顺手检查这个 token 是否真的能访问 Synapse Admin API：

TOKEN_FILE="/root/.config/synapse-admin/access_token"
TOKEN="$(cat "$TOKEN_FILE")"

curl -sS -o /tmp/synapse-admin-token-check.json -w 'http_code=%{http_code}\n' \
  -H "Authorization: Bearer ${TOKEN}" \
  'http://127.0.0.1:8008/_synapse/admin/v1/statistics/users/media?limit=1'

cat /tmp/synapse-admin-token-check.json
rm -f /tmp/synapse-admin-token-check.json

如果输出是：

http_code=200

并且后面有 users、total 这类字段，说明 token 有效，而且账号有 Admin API 权限。

如果输出是：

http_code=401

并且 JSON 里有 M_UNKNOWN_TOKEN，说明 token 已失效，需要重新登录这个专用管理员账号，生成新的 access token。

如果输出是：

http_code=403

通常说明 token 本身能识别，但这个用户不是 Synapse 管理员，不能调用 Admin API。

八、手动预演一次分层清理

先手动跑一遍，确认 Admin API、token、本机地址和分层规则都正常。本文最终规则是：

5MiB 以内文件长期保留。
超过 5MiB 的文件，90 天后清理。
超过 100MiB 的超大文件 7 天后清理。
远程媒体缓存 7 天后清理。

先查看会进入分层清理范围的本地媒体：

docker exec postgres16 psql -U synapse_user -d synapse -P pager=off -c "
  select media_id,
         user_id,
         pg_size_pretty(media_length::bigint) as size,
         media_length,
         media_type,
         to_timestamp(created_ts/1000) as created,
         to_timestamp(last_access_ts/1000) as last_access
    from local_media_repository
   where media_length > 5242880
   order by media_length desc
   limit 30;"

这张表能让你预览哪些媒体超过 5MiB。加密房间里的视频、语音或文件在 Synapse 里通常都会显示为 application/octet-stream，服务端无法直接判断它原本是什么类型，所以规则只能按大小和时间执行。

手动执行三条清理规则：

TOKEN="$(cat /root/.config/synapse-admin/access_token)"
LARGE_BEFORE_TS="$(python3 - <<'PY'
import time
print(int((time.time() - 7 * 86400) * 1000))
PY
)"
MEDIUM_BEFORE_TS="$(python3 - <<'PY'
import time
print(int((time.time() - 90 * 86400) * 1000))
PY
)"
REMOTE_BEFORE_TS="$(python3 - <<'PY'
import time
print(int((time.time() - 7 * 86400) * 1000))
PY
)"

curl -sS -X POST \
  -H "Authorization: Bearer ${TOKEN}" \
  "http://127.0.0.1:8008/_synapse/admin/v1/media/delete?before_ts=${LARGE_BEFORE_TS}&size_gt=104857600&keep_profiles=true"

curl -sS -X POST \
  -H "Authorization: Bearer ${TOKEN}" \
  "http://127.0.0.1:8008/_synapse/admin/v1/media/delete?before_ts=${MEDIUM_BEFORE_TS}&size_gt=5242880&keep_profiles=true"

curl -sS -X POST \
  -H "Authorization: Bearer ${TOKEN}" \
  "http://127.0.0.1:8008/_synapse/admin/v1/purge_media_cache?before_ts=${REMOTE_BEFORE_TS}"

普通聊天文件命中规则后会清理。keep_profiles=true 是为了尽量保护仍被用户头像、房间头像等 profile image 使用的媒体，不是为了保护普通大文件。

九、写成分层定期脚本

创建脚本目录：

mkdir -p /root/data/docker_data/synapse/scripts
vi /root/data/docker_data/synapse/scripts/purge-staged-media.sh

脚本内容：

#!/bin/sh
set -eu

SYNAPSE_URL="${SYNAPSE_URL:-http://127.0.0.1:8008}"
TOKEN_FILE="${TOKEN_FILE:-/root/.config/synapse-admin/access_token}"
REMOTE_DAYS="${REMOTE_DAYS:-7}"
KEEP_PROFILES="${KEEP_PROFILES:-true}"
REPORT_LIMIT="${REPORT_LIMIT:-30}"
SMALL_KEEP_SIZE="${SMALL_KEEP_SIZE:-5242880}"
MEDIUM_DAYS="${MEDIUM_DAYS:-90}"
MEDIUM_SIZE_GT="${MEDIUM_SIZE_GT:-5242880}"
LARGE_DAYS="${LARGE_DAYS:-7}"
LARGE_SIZE_GT="${LARGE_SIZE_GT:-104857600}"

TOKEN="$(cat "$TOKEN_FILE")"
TMP="/tmp/synapse-media-tiered-purge-response.json"

before_ms() {
  python3 - "$1" <<'PY'
import sys
import time
days = int(sys.argv[1])
print(int((time.time() - days * 86400) * 1000))
PY
}

api_post() {
  label="$1"
  url="$2"
  echo "[$(date '+%Y-%m-%dT%H:%M:%S%z')] ${label}"
  http_code="$(curl -sS -o "$TMP" -w '%{http_code}' -X POST -H "Authorization: Bearer ${TOKEN}" "$url")"
  cat "$TMP"
  echo
  if [ "$http_code" -lt 200 ] || [ "$http_code" -ge 300 ]; then
    echo "${label}: Admin API failed with HTTP ${http_code}" >&2
    rm -f "$TMP"
    exit 1
  fi
  rm -f "$TMP"
}

report_local_candidates() {
  echo "[$(date '+%Y-%m-%dT%H:%M:%S%z')] local media policy report"
  echo "Policy: <=5MiB kept long-term; >5MiB deleted after ${MEDIUM_DAYS}d; >100MiB deleted after ${LARGE_DAYS}d; remote cache deleted after ${REMOTE_DAYS}d."
  docker exec postgres16 psql -U synapse_user -d synapse -P pager=off -c "
    select media_id,
           user_id,
           pg_size_pretty(media_length::bigint) as size,
           media_length,
           media_type,
           to_timestamp(created_ts/1000) as created,
           to_timestamp(last_access_ts/1000) as last_access
      from local_media_repository
     where media_length > ${SMALL_KEEP_SIZE}
     order by media_length desc
     limit ${REPORT_LIMIT};"
}

purge_local_media() {
  label="$1"
  days="$2"
  size_gt="$3"
  before_ts="$(before_ms "$days")"
  api_post "$label older_than=${days}d size_gt=${size_gt} keep_profiles=${KEEP_PROFILES}" \
    "${SYNAPSE_URL}/_synapse/admin/v1/media/delete?before_ts=${before_ts}&size_gt=${size_gt}&keep_profiles=${KEEP_PROFILES}"
}

purge_remote_cache() {
  before_ts="$(before_ms "$REMOTE_DAYS")"
  api_post "purge remote media cache older_than=${REMOTE_DAYS}d" \
    "${SYNAPSE_URL}/_synapse/admin/v1/purge_media_cache?before_ts=${before_ts}"
}

report_local_candidates
purge_local_media "purge local large media" "$LARGE_DAYS" "$LARGE_SIZE_GT"
purge_local_media "purge local medium media" "$MEDIUM_DAYS" "$MEDIUM_SIZE_GT"
purge_remote_cache

授权：

chmod 700 /root/data/docker_data/synapse/scripts/purge-staged-media.sh

先手动跑一次：

/root/data/docker_data/synapse/scripts/purge-staged-media.sh

正常输出会包含本地大媒体列表、本地媒体删除结果、远程缓存删除结果，例如：

purge local large media older_than=7d size_gt=104857600 keep_profiles=true
{"deleted_media":[],"total":0}
purge local medium media older_than=90d size_gt=5242880 keep_profiles=true
{"deleted_media":[],"total":0}
purge remote media cache older_than=7d
{"deleted":0}

再看体积：

du -h -d 1 /root/data/docker_data/synapse/data/media_store | sort -h

十、加入 cron 定时任务

Alpine 上可以用 root 的 crontab：

crontab -e

每天凌晨 4:23 执行：

23 4 * * * /root/data/docker_data/synapse/scripts/purge-staged-media.sh 2>&1 | logger -t synapse-media-purge

确认 crond 启动：

rc-service crond status
rc-update add crond default
rc-service crond start

以后看日志：

logread | grep synapse-media-purge

如果系统没有 logread，可以临时把 cron 输出写到文件，但要配合日志轮转，避免这个日志文件反过来变成新问题。

十一、按用户定位媒体占用

如果 media_store 突然变大，先查每个本地用户的媒体占用。这个接口适合回答“到底是谁上传了大文件”，不等于查出来就可以自动删除。

TOKEN="$(cat /root/.config/synapse-admin/access_token)"

curl -sS \
  -H "Authorization: Bearer ${TOKEN}" \
  "http://127.0.0.1:8008/_synapse/admin/v1/statistics/users/media?limit=100&order_by=media_length&dir=b"

常见返回会包含每个用户上传媒体数量和总大小。dir=b 表示倒序，通常能把占用最大的用户排在前面。

想看某个用户具体上传了哪些媒体，可以查用户媒体列表。因为 Matrix 用户 ID 里有 @ 和 :，URL 里要先做编码：

USER_ID='@alice:666666.xyz'
ENCODED_USER_ID="$(USER_ID="$USER_ID" python3 - <<'PY'
import os
import urllib.parse
print(urllib.parse.quote(os.environ["USER_ID"], safe=""))
PY
)"

curl -sS \
  -H "Authorization: Bearer ${TOKEN}" \
  "http://127.0.0.1:8008/_synapse/admin/v1/users/${ENCODED_USER_ID}/media?limit=20&order_by=media_length&dir=b"

如果你明确知道某个临时测试账号上传的媒体都可以删，可以按用户删除。先谨慎限制数量，例如每次 50 个：

curl -sS -X DELETE \
  -H "Authorization: Bearer ${TOKEN}" \
  "http://127.0.0.1:8008/_synapse/admin/v1/users/${ENCODED_USER_ID}/media?limit=50&order_by=media_length&dir=b"

这个按用户删除的接口只适合清理测试账号、临时导入账号或你已经确认消息引用都可以失效的误传账号。它不适合直接套到正常聊天用户上，也不应该放进 cron。正常聊天里的本地上传媒体，仍然要先确认对应消息已经撤回或不再可见。

十二、分阶段上线

第一次上线不要直接交给 cron 长期跑。推荐分三步：

第一阶段：只手动执行

/root/data/docker_data/synapse/scripts/purge-staged-media.sh

看本地大媒体列表、本地媒体删除结果和远程缓存清理结果，再看目录体积：

du -h -d 1 /root/data/docker_data/synapse/data/media_store | sort -h

第二阶段：每天跑，但只观察日志

加入 cron 后先观察一周：

logread | grep synapse-media-purge

重点看有没有异常返回，比如 token 失效、接口 404、网络连接失败。

第三阶段：作为常态维护

确认没有异常后，长期保留这组默认值：

REMOTE_DAYS=7
REPORT_LIMIT=30
SMALL_KEEP_SIZE=5242880
MEDIUM_DAYS=90
MEDIUM_SIZE_GT=5242880
LARGE_DAYS=7
LARGE_SIZE_GT=104857600
KEEP_PROFILES=true

也就是：

<=5MiB：长期保留
>5MiB：90 天后清理
>100MiB：7 天后清理
远程缓存：7 天后清理

这样能保护语音、小图和普通小文件，同时让大视频、大压缩包和其他大文件按确定规则到期清理。加密房间里的过期大文件清理后，旧消息点开可能不可用，这是控制硬盘长期增长需要提前告知用户的取舍。

一次生产环境实测结果类似：

Policy: <=5MiB kept long-term; >5MiB deleted after 90d; >100MiB deleted after 7d; remote cache deleted after 7d.
purge local large media older_than=7d size_gt=104857600 keep_profiles=true
{"deleted_media":[],"total":0}
purge local medium media older_than=90d size_gt=5242880 keep_profiles=true
{"deleted_media":[],"total":0}
purge remote media cache older_than=7d
{"deleted":0}
crond 状态：started

这里要注意：Admin API 的用户媒体统计和当前磁盘目录大小不一定完全相等。比如之前手动清过文件、迁移过目录，或者数据库里还有历史媒体记录时，你可能会看到 statistics/users/media 里仍有几百 MB 的 media_length，但 du 查看 media_store 已经很小。判断当前磁盘压力时，还是以 du -h -d 1 /root/data/docker_data/synapse/data/media_store 为准。

十三、不要把清空 media_store 当维护手段

直接清空 media_store 会制造大量旧附件 404，也可能影响头像和房间头像。它只能作为“磁盘马上写满、服务已经受影响、并且你明确接受所有旧聊天附件都可能打不开”的最后救急方案，不是日常维护方式。

真正生产维护时，不要直接 rm -rf 本地媒体目录。这个动作不会替你处理房间里的旧消息引用，只会让客户端以后在打开旧附件时遇到缺失文件。

如果已经到了必须救急的程度，至少先停 Synapse、做快照或备份、把要清理的目录移动到 quarantine，观察一段时间后再删除。流程示例：

先停 Synapse：

cd /root/data/docker_data/synapse
docker compose stop synapse

移动高风险目录到 quarantine，而不是直接硬删：

mkdir -p /root/data/docker_data/synapse/data/media_store_quarantine
mv /root/data/docker_data/synapse/data/media_store/local_content \
   /root/data/docker_data/synapse/data/media_store/local_thumbnails \
   /root/data/docker_data/synapse/data/media_store_quarantine/
mkdir -p /root/data/docker_data/synapse/data/media_store/local_content \
         /root/data/docker_data/synapse/data/media_store/local_thumbnails

确认目录权限：

chmod 777 /root/data/docker_data/synapse/data/media_store \
          /root/data/docker_data/synapse/data/media_store/local_content \
          /root/data/docker_data/synapse/data/media_store/local_thumbnails \
          /root/data/docker_data/synapse/data/media_store/url_cache \
          /root/data/docker_data/synapse/data/media_store/url_cache_thumbnails

启动 Synapse：

docker compose up -d synapse
curl -i http://127.0.0.1:8008/_matrix/client/versions

这种做法的代价仍然很大：

  • 旧消息里的图片、视频、文件可能打不开。
  • 头像或房间头像如果刚好在本地媒体里，可能需要重新设置。
  • 数据库里可能仍有旧媒体记录，但文件已经不存在。

所以它只适合一次性救急，不适合作为日常维护手段。日常维护应优先限制上传大小，并使用前面的分层脚本自动清理到期大文件和远程缓存。

十四、孤立文件先不做硬删

孤立文件指的是：磁盘上还有文件，但数据库里已经找不到对应媒体记录。它确实可以清理，但第一版不建议“检测到就即时删除”。

更稳的节奏是：

第一阶段：只生成报告，不移动、不删除
第二阶段：确认报告合理后，移动到 quarantine
第三阶段：quarantine 保留 7 天后再删除

原因很简单：孤立文件判断依赖脚本对 Synapse 数据库、缩略图、远程缓存、URL 预览目录的理解。一旦脚本误判，硬删以后不好恢复。隔离不是为了等用户报错，而是为了让管理员能看到脚本动了哪些文件、多少体积，并且还有回滚空间。

如果只是当前这个不到 3GB 的 media_store，优先实施分层脚本就够了。孤立文件检测可以以后作为独立维护脚本再加，不需要和第一版混在一起。

十五、推荐最终配置

对个人 Synapse，我会这样配置：

max_upload_size: 20M

NPM Advanced：

client_max_body_size 20M;

如果你的需求是“允许用户继续发语音、小图、小文件，同时控制大文件长期占用”，脚本默认值保持：

REMOTE_DAYS=7
REPORT_LIMIT=30
SMALL_KEEP_SIZE=5242880
MEDIUM_DAYS=90
MEDIUM_SIZE_GT=5242880
LARGE_DAYS=7
LARGE_SIZE_GT=104857600
KEEP_PROFILES=true

然后每天跑一次 cron：

23 4 * * * /root/data/docker_data/synapse/scripts/purge-staged-media.sh 2>&1 | logger -t synapse-media-purge

这样会长期保留 5MiB 以内文件；超过 5MiB 的文件 90 天后清理；超过 100MiB 的超大文件 7 天后清理；远程缓存 7 天后清理。

十六、用户告知单独处理

分层清理上线后，最好主动告诉用户：5MiB 以内文件长期保留；超过 5MiB 的文件，90 天后清理；超过 100MiB 的超大文件 7 天后清理；请自行保存重要大文件。

用户告知建议用 Synapse Server Notices 单独做。完整教程已经拆到另一篇：
