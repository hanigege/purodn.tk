烧鸡商城 Docker 手动安装与入口说明

记录烧鸡商城源码位置、第三方机器手动部署步骤、首次初始化入口、默认访问链接，以及短信和支付上线前的接入事项。

这篇是烧鸡商城的部署备忘。适合以后把系统迁移到第三方机器、重装服务器、或者交给别人手动安装时照着做。

当前 Hermes 机器上的源码目录：

/home/hermesuser/shaoji-mall

也可以写成：

~/shaoji-mall

线上 GitHub 仓库：

git@github.com:hanigege/shaoji.git

系统组成

源码主要分成这几块：

路径 说明
frontend/ 用户商城、管理后台、配送员端的前端页面
backend/ FastAPI 后端接口
sql/init.sql MySQL 初始化建表脚本
docker-compose.yml MySQL、Redis、后端、前端容器编排
uploads/ 上传文件目录，前后端容器都会挂载

容器默认名称：

容器 说明
shaoji-frontend 前端 Nginx 容器
shaoji-backend 后端 FastAPI 容器
shaoji-mysql MySQL 8.0
shaoji-redis Redis 7

默认端口：

端口 用途
3000 前端入口
8000 后端 API
3306 MySQL
6379 Redis

生产环境建议只对外开放前端入口，后端、数据库、Redis 尽量不要直接暴露到公网。

第三方机器安装前准备

第三方机器建议是 Linux 服务器，先安装：

  • Git
  • Docker
  • Docker Compose

Ubuntu / Debian 可以参考：

sudo apt update
sudo apt install -y git ca-certificates curl

Docker 可以按官方文档安装。安装后确认：

docker --version
docker compose version

如果当前用户没有 Docker 权限，可以加入 docker 组：

sudo usermod -aG docker $USER

然后重新登录一次 SSH。

手动安装步骤

拉取源码：

cd ~
git clone git@github.com:hanigege/shaoji.git shaoji-mall
cd ~/shaoji-mall

如果第三方机器还没有配置 GitHub SSH key，可以临时用 HTTPS：

git clone https://github.com/hanigege/shaoji.git shaoji-mall

建议新建 .env 文件，至少改掉默认密钥和数据库密码：

cd ~/shaoji-mall
nano .env

示例：

MYSQL_ROOT_PASSWORD=请改成强密码
MYSQL_DATABASE=shaoji_mall
MYSQL_USER=shaoji
MYSQL_PASSWORD=请改成强密码
SECRET_KEY=请改成随机长字符串
APP_ENV=production
CORS_ORIGINS=https://你的域名
ENABLE_MOCK_PAYMENT=false
MAX_UPLOAD_MB=5

ALIYUN_ACCESS_KEY_ID=
ALIYUN_ACCESS_KEY_SECRET=
ALIYUN_SIGN_NAME=
ALIYUN_TEMPLATE_CODE=

KUAIDI100_KEY=
KUAIDI100_CUSTOMER=

启动：

cd ~/shaoji-mall
docker compose up -d --build

查看状态：

docker ps

健康检查：

curl http://127.0.0.1:8000/health

正常会返回：

{"status":"ok"}

安装后的默认入口

假设服务器 IP 是 服务器IP，默认入口如下：

页面 地址
商城首页 http://服务器IP:3000/
用户登录 http://服务器IP:3000/login
我的订单 http://服务器IP:3000/orders
个人中心 http://服务器IP:3000/profile
首次初始化 http://服务器IP:3000/init
管理后台登录 http://服务器IP:3000/admin/login
管理后台首页 http://服务器IP:3000/admin/dashboard
配送员登录 http://服务器IP:3000/deliverer/login
配送员订单 http://服务器IP:3000/deliverer/orders
后端健康检查 http://服务器IP:8000/health

注意：后台登录入口可以在管理后台的系统设置里自定义。改完以后，例如设置成 manage2026，后台登录入口就变成：

http://服务器IP:3000/manage2026/login

原来的 /admin/login 会被拦截，不再作为有效登录入口。

默认管理账号和密码

系统不建议内置固定默认账号，例如 admin/admin123 这种很容易被扫。

刚安装好后，应访问：

http://服务器IP:3000/init

在初始化向导里自己设置：

  • 管理员手机号
  • 管理员密码
  • 管理员名称
  • 店铺名称
  • 联系电话
  • 店铺地址

初始化完成后，再用你设置的手机号和密码登录后台。

如果忘记管理员密码，可以在已有管理员后台里重置；如果已经完全进不去，再走数据库或后端脚本进行安全重置。

常用运维命令

进入项目目录：

cd ~/shaoji-mall

查看容器：

docker ps

查看后端日志：

docker compose logs --tail=200 backend

查看模拟短信验证码日志：

docker compose logs --tail=200 backend | grep SMS

注意：必须在 ~/shaoji-mall 目录下执行 docker compose，否则会出现：

no configuration file provided: not found

如果不想切目录，也可以指定 compose 文件：

docker compose -f ~/shaoji-mall/docker-compose.yml logs --tail=200 backend

重启后端：

docker compose restart backend

重新构建并发布前端：

cd ~/shaoji-mall/frontend
npm run build

cd ~/shaoji-mall
docker compose build frontend
docker compose up -d frontend

完整重建：

cd ~/shaoji-mall
docker compose up -d --build

当前短信验证状态

目前短信接口还是 Mock 模式：系统会生成 6 位验证码，但不会真正发短信，验证码会打印在后端日志里。

查看方式：

cd ~/shaoji-mall
docker compose logs --tail=200 backend | grep SMS

用户登录页已经有拖拽形状验证，并且拖拽验证会绑定手机号：

  • 必须先输入完整手机号
  • 拖拽验证和当前手机号绑定
  • 手机号改了以后验证立即失效
  • 后端也会校验手机号绑定关系
  • A 手机号通过的验证不能拿去给 B 手机号发验证码

正式上线前，需要接真实短信服务，例如阿里云短信或腾讯云短信。通常要准备：

  • 企业或个人主体资质
  • 短信签名审核
  • 短信模板审核
  • AccessKey 或密钥
  • 短信发送频率限制
  • 短信费用余额监控

.env 里预留了阿里云短信配置：

ALIYUN_ACCESS_KEY_ID=
ALIYUN_ACCESS_KEY_SECRET=
ALIYUN_SIGN_NAME=
ALIYUN_TEMPLATE_CODE=

当前支付状态

目前支付流程还是 Mock 模式，适合开发测试，不是真实收款。

2026-07-06 已做过一次生产安全加固：真实支付没有接入验签前，支付回调接口会直接拒绝处理，不会再因为别人伪造回调就把订单改成已支付。

也就是说，在正式接微信支付或支付宝之前：

  • 开发测试可以继续用 Mock 支付。
  • 真正上线时必须设置 ENABLE_MOCK_PAYMENT=false。
  • 真实支付回调必须完成平台签名校验、商户号校验、订单金额校验后，才能允许改订单状态。

正式上线前，需要接入第三方支付，例如：

  • 微信支付
  • 支付宝

通常要准备：

  • 商户号
  • 应用 ID
  • API 密钥 / 证书
  • 支付回调地址
  • 退款回调地址
  • 服务器公网域名
  • HTTPS 证书
  • 主体资料、备案或平台要求的资质

建议等营业主体、域名、备案、商户号都准备好以后，再接真实支付。这样不会因为资料没齐反复改配置。

快递和配送

系统支持两类履约方式：

  • 同城配送：管理员派单，配送员接单/配送，用户订单里可看到配送员姓名和联系电话。
  • 快递邮寄：管理员发货，填写物流公司和快递单号，用户侧可查看物流信息。

快递查询预留了快递 100 配置：

KUAIDI100_KEY=
KUAIDI100_CUSTOMER=

没有配置时，快递查询接口不会返回真实物流轨迹。

2026-07-06 安全加固记录

这次已经同步到 GitHub：

9acb754 fix: harden production security defaults
020b480 fix: run backend without reload in production

上线前备份位置：

/home/hermesuser/shaoji-backups/shaoji-mall-worktree-20260706_112222-before-security-hardening.tar.gz

本次修复内容：

  • 支付回调：未接入真实支付验签前，任何 /api/pay/notify/* 回调都不会改订单状态，会直接拒绝。
  • 模拟支付：加了开关，生产环境可以强制关闭。
  • 商品详情富文本：保留 Markdown/富文本展示，但先做白名单清洗，降低 XSS 风险。
  • 图片上传：限制大小，校验图片类型，防止异常大图或非图片文件压垮服务。
  • 物流轨迹接口：加了权限校验，用户只能看自己的订单，配送员只能看自己负责的订单，管理员可以看全部。
  • 配送员重置密码：去掉固定 admin123，改成每次生成随机临时密码。
  • 后端错误返回：不再把内部异常原文直接返回给前端。
  • CORS：改成配置化，生产环境不允许使用 * 通配。
  • 端口暴露：MySQL、Redis、后端 API 都只绑定本机，外部只留前端入口。
  • 后端启动：去掉 Docker 里的 --reload 开发模式，改为生产启动。

当前为了方便测试，服务器仍按 development 跑。真正上线时，.env 至少要这样设置：

APP_ENV=production
SECRET_KEY=一串足够长的随机密钥
CORS_ORIGINS=https://你的正式域名
ENABLE_MOCK_PAYMENT=false
MAX_UPLOAD_MB=5

生产环境启动时，后端会自动检查：

  • CORS_ORIGINS 不能是 *
  • SECRET_KEY 不能是默认值
  • ENABLE_MOCK_PAYMENT 不能为 true

如果这些没配好，服务会拒绝启动，这是故意的，避免忘记改配置就上线。

数据备份建议

上线前一定要做数据库备份策略。

手动备份 MySQL：

cd ~/shaoji-mall
docker exec shaoji-mysql mysqldump -uroot -p"$MYSQL_ROOT_PASSWORD" shaoji_mall > backup_$(date +%Y%m%d_%H%M%S).sql

如果这个命令拿不到宿主机环境变量，可以进入容器查看或使用 .env 里的 root 密码。

同时建议备份：

  • MySQL 数据
  • uploads/ 上传文件
  • .env 配置文件
  • 当前 Git commit 号

更新代码

以后从 GitHub 更新：

cd ~/shaoji-mall
git pull
docker compose up -d --build

如果只改了后端 Python 代码，因为后端挂载了 backend/app，通常重启后端即可：

docker compose restart backend

如果改了前端 Vue 代码，需要重新构建前端：

cd ~/shaoji-mall/frontend
npm run build
cd ~/shaoji-mall
docker compose build frontend
docker compose up -d frontend
