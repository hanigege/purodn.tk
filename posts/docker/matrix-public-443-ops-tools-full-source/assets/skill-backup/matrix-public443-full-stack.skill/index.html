Matrix Public 443 Full Stack

Use this for the user’s Matrix VPS pattern: Alpine/OpenRC, normal SSH user such as jay, root via sudo -i or sudo -n sh -c, no root password, production runtime under /root/data/docker_data.

Hard Rules

  • Never print real passwords, access tokens, TURN secrets, LiveKit secrets, Synapse registration/shared secrets, signing keys, database passwords, or bcrypt hashes in chat or public docs.
  • Treat /root/data/docker_data as the only production runtime tree. Do not introduce normal-user home-directory template steps into beginner install docs.
  • Keep secrets under /root/data/docker_data/_secrets, mode 700 for directories and 600 for files.
  • Do not copy PostgreSQL data, media store, signing keys, Caddy runtime data, logs, SQLite DBs, caches, or _secrets contents into Hugo/articles/source bundles.
  • For media deletion, keep explicit Chinese comments near the deletion boundary. Default keep_profiles=true; it preserves Synapse profile-referenced media such as user avatars and room avatars, not ordinary chat attachments.
  • Public access to high-risk tooling must go through Caddy Basic Auth and local-only backend listeners.

matrix-admin-tools boundaries:

  • Backend listens only on 127.0.0.1:8190.
  • Public domain is matrix-tools.jgaga.tk behind Caddy basic_auth.
  • /notice may be a direct browser entry after Basic Auth succeeds, but unauthenticated public requests must still return 401; never bypass Caddy Basic Auth for the notice form.
  • GET /notice should only render the Server Notice form. Actual sending must remain POST /notice to prevent link-open side effects.
  • Reads only /root/data/docker_data/_secrets/synapse-admin/access_token.
  • Does not store tokens in app DB/logs/docs/Caddyfile.
  • Daily media cleanup checks rules; it must not be described as deleting everything every day.

matrix-musicbot boundaries:

  • Use upstream image ghcr.io/sultanalburaq/matrix-element-call-musicbot:latest unless the user asks otherwise.
  • Writable dirs logs, data, cache should be owned by UID 10001.
  • Explain that !join makes the bot join the call; the human still clicks Element’s “加入”.

matrix-bot-sync and matrix-bot-weather boundaries:

  • The current production bot stack is not a generic matrix-simple-bot; it uses matrix-bot-sync, matrix-bot-weather, and matrix-bot-monitor scripts under /root/data/docker_data/synapse/scripts/.
  • OpenRC services are /etc/init.d/matrix-bot-sync, /etc/init.d/matrix-bot-weather, and /etc/init.d/matrix-bot-monitor; enable these three in default and verify with rc-status default.
  • matrix-bot-sync handles invites and chat commands. matrix-bot-weather sends scheduled forecasts. Do not merge them into cron unless the user explicitly asks; production uses OpenRC loops.
  • Bot token and password live under /root/data/docker_data/_secrets/matrix-bot/. Never print them in final answers.

synapse-media-purge boundaries:

  • Production has /etc/init.d/synapse-media-purge and /root/data/docker_data/synapse/scripts/run-synapse-media-purge-loop.sh, but it is not currently enabled in the default runlevel. Preserve the file during disaster recovery; enable it only after confirming the user wants automatic media cleanup active.
  • Media cleanup is a destructive production boundary. Keep Chinese comments near cleanup scripts explaining daily-once stamps, keep_profiles=true, and why remote cache and local media policies are separate.
  • Default staged cleanup uses /root/data/docker_data/synapse/scripts/purge-staged-media.sh; the older purge-large-media.sh is retained as a legacy single-policy entry, not the default path.

matrix-bot-monitor boundaries:

  • Production monitor config lives at /root/data/docker_data/synapse/scripts/matrix-bot-monitor.json.
  • For the current jgaga.tk public-443 stack, target_users must include exactly @black:jgaga.tk, @honey:jgaga.tk, and @admin2:jgaga.tk unless the user explicitly changes the alert recipient list. Do not leave it at only @admin:jgaga.tk.
  • Keep expected_containers aligned with current docker ps; the public-443 stack uses element-call, not old web-call.
  • After changing monitor recipients or runtime checks, run /root/data/docker_data/synapse/scripts/matrix-bot-monitor.py --test-alert and confirm the state file has direct_rooms for every target user before declaring alerts fixed.

Production-Calibrated Replication Checklist

When the user gives a new public VPS and asks to replicate the current production stack, calibrate against the live 192.227.236.209 layout or the private repo hanigege/matrix-synapse-public443-jgaga.tk before declaring the skill sufficient. A complete replication includes all of these runtime surfaces:

  • Docker containers: caddy, matrix-synapse, postgres16, element-web, element-call, livekit-server-test, matrix-livekit-jwt-service, coturn, ntfy, synapse-admin, matrix-admin-tools, and matrix-musicbot.
  • Root runtime files under /root/data/docker_data, not normal-user home templates.
  • OpenRC services: matrix-bot-sync, matrix-bot-weather, matrix-bot-monitor, and preserved synapse-media-purge.
  • Secrets under /root/data/docker_data/_secrets, plus Synapse signing key when keeping the same Matrix identity.
  • Caddy public 80/443 routes, including matrix-tools.jgaga.tk Basic Auth and /notice behavior.
  • Bot runtime configs: monitor targets, weather config, MusicBot config, radio aliases, podcast aliases, and patched bot file when present.

For a different domain, replace domain names consistently across Caddy, Synapse server_name and public_baseurl, Element Web config, Element Call config, LiveKit/JWT config, coturn realm, bot MXIDs, and monitor target MXIDs. Do not blindly keep jgaga.tk unless the user is intentionally preserving the same Matrix identity.
