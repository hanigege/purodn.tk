sing-box 生产机纯手动备份与恢复

不用 GitHub 仓库，只拿生产机备份包，在第三方 Linux 新机器上手动恢复 sing-box 网关、Rule UI、TProxy、DNS 53 端口和 systemd 服务。

这篇文章记录的是一次真实生产机的手动备份和恢复方案。目标很简单：不依赖 GitHub 仓库，不跑一键安装脚本，只拿一份备份包，在第三方新机器上尽量完整恢复 sing-box 网关、Rule UI、TProxy、DNS 53 端口和 systemd 服务。

本文提供一份从当前生产机 10.20.20.6 提取、再本地脱敏后的恢复包。它保留生产机实际运行的 Rule UI、helper 脚本、systemd 单元、规则集、TProxy/sysctl 文件、Clash UI 静态文件和仓库验证过的 sing-box 二进制；同时移除了真实代理节点、Rule UI token、Clash secret、生产 IPv6 FakeIP 前缀、生产自定义规则列表、运行缓存和历史 .bak/backups 文件。

下载文件：

  • sing-box-production-manual-restore-20260611.tar.gz
  • sing-box-production-manual-restore-20260611.tar.gz.sha256

校验哈希：

eceee77d1ef1a515fd25235ff85ca9cab0a4d895c403c407a64d64ed71949358  sing-box-production-manual-restore-20260611.tar.gz

这份脱敏包恢复后的初始状态和 GitHub 仓库一键安装的安全默认值保持一致：只有 TEMPLATE-HY2、TEMPLATE-VLESS 两个模板节点，自定义白名单/黑名单/灰名单/DDNS 规则为空，Rule UI token 是 change-me-rule-ui-token，Clash secret 是 change-me-clash-secret。启动前或启动后必须把这些占位值换成自己的真实值。

当前主仓库已经固定启用 FakeIP QUIC 保护：系统只拦截发往 FakeIP 网段的 UDP/443，让浏览器回落到 TCP，减少 QUIC 长连接占满代理带宽和连接表；真实游戏/语音 UDP 不受影响。这个策略属于网关稳定性边界，不再在 UI 中提供关闭入口。

本文这份包已经在一台干净的 Ubuntu 25.04 amd64 测试机上按下文流程恢复通过。测试机地址为 10.20.20.16，网卡为 eth0，IPv6 DNS 监听地址为 fd88::1616；最终 sing-box、sing-box-tproxy、singbox-rule-ui、规则更新 timer 均为 active，53/9090/9091/9888 端口正常监听。

在新机器上准备恢复包

把下载好的备份包和校验文件放到新机器的 /root/，例如：

cd /root
ls -lh sing-box-production-manual-restore-20260611.tar.gz*
sha256sum -c sing-box-production-manual-restore-20260611.tar.gz.sha256

看到 OK 再解压：

cd /root
tar -xzpf sing-box-production-manual-restore-20260611.tar.gz
cd sing-box-production-manual-restore-20260611

先看清包里的路径，不要急着覆盖：

find files -maxdepth 4 -type f | sort | sed -n '1,120p'
sed -n '1,120p' manifest/restore-package.txt 2>/dev/null || true
sed -n '1,120p' README-BACKUP.txt 2>/dev/null || true

包内通常按 files/ 保存原机器路径，核心内容包括：

files/etc/sing-box/
files/opt/singbox-rule-ui/
files/usr/local/bin/sing-box
files/usr/local/bin/sing-box-gateway-info
files/usr/local/bin/sing-box-gateway-uninstall
files/usr/local/sbin/refresh-sing-box-runtime-config
files/usr/local/sbin/sing-box-tproxy-setup
files/usr/local/sbin/update-sing-box-rules-jsdelivr
files/etc/systemd/system/sing-box.service
files/etc/systemd/system/sing-box-tproxy.service
files/etc/systemd/system/singbox-rule-ui.service
files/etc/systemd/system/update-sing-box-rules-jsdelivr.service
files/etc/systemd/system/update-sing-box-rules-jsdelivr.timer
files/etc/sysctl.d/99-sing-box-tproxy.conf
manifest/
README-BACKUP.txt

/etc/sing-box/ 是运行配置、管理 JSON、规则集、UI token 和 Clash UI 静态文件；/opt/singbox-rule-ui/ 是 Rule UI 后端和前端；/usr/local/bin/ 里有 sing-box 和信息/卸载辅助命令；/usr/local/sbin/ 里的脚本负责运行时重渲染、TProxy/nftables 和规则更新；systemd 单元和 sysctl 文件决定服务启动顺序与 TProxy 内核参数。

适用的新机器

恢复包里的 sing-box 二进制来自生产机，只能用于同架构机器。先在原机器或备份 manifest 里确认：

sing-box version

新机器建议满足：

  • Linux，架构要和备份包里的 /usr/local/bin/sing-box 一致。
  • 使用 systemd。
  • 有 root 权限。
  • 已安装 python3、iproute2、nftables、tar、coreutils。
  • 网卡名、LAN IPv4、LAN IPv6 和生产机一致时恢复最省事；如果不一致，要按后文修改。

Ubuntu/Debian 可以先装基础依赖：

apt update
apt install -y python3 iproute2 nftables tar coreutils

如果是 ARM/aarch64 新机器，不要直接用 amd64 备份包里的 /usr/local/bin/sing-box。配置和 UI 可以参考恢复，但 sing-box 二进制必须换成同版本、同配置兼容的 ARM 版本，再重新 sing-box check。

先检查并改掉敏感值

这份公开包已经把真实节点、Rule UI token 和 Clash secret 替换成占位值。恢复到新机器前，至少先确认这些文件：

sed -n '1,160p' /root/sing-box-production-manual-restore-20260611/files/etc/sing-box/manager/nodes.json
sed -n '1,120p' /root/sing-box-production-manual-restore-20260611/files/etc/sing-box/manager/groups.json
cat /root/sing-box-production-manual-restore-20260611/files/etc/sing-box/rule-ui/token

如果你想先恢复 UI，再通过网页改节点，可以先保留包里的模板节点，等服务启动后在 Rule UI 里删除或改成新机器要使用的真实节点。

如果你要在恢复前直接改文件，先编辑：

nano /root/sing-box-production-manual-restore-20260611/files/etc/sing-box/manager/nodes.json
nano /root/sing-box-production-manual-restore-20260611/files/etc/sing-box/manager/groups.json
nano /root/sing-box-production-manual-restore-20260611/files/etc/sing-box/manager/base.json

nodes.json 里默认只有模板节点；groups.json 里默认节点指向 TEMPLATE-HY2；base.json 里确认 LAN IPv4 是否要换成新机器地址。Rule UI token 和 Clash secret 必须改掉：

printf '%s\n' '换成你自己的-Rule-UI-token' > /root/sing-box-production-manual-restore-20260611/files/etc/sing-box/rule-ui/token
# Clash secret 在 base.json/config.json 的 experimental.clash_api.secret 里，恢复后也可以在 UI 或文件里改。

停掉可能冲突的服务

如果新机器以前装过 sing-box 或其它 DNS 服务，先停掉。没有这些服务时命令报错可以忽略：

systemctl stop singbox-rule-ui.service 2>/dev/null || true
systemctl stop sing-box.service 2>/dev/null || true
systemctl stop sing-box-tproxy.service 2>/dev/null || true

检查 53 端口是谁占着：

ss -ltnup | grep -E ':53\b' || true

如果看到 systemd-resolved 占用 127.0.0.53:53 或 0.0.0.0:53，按下面处理：

mkdir -p /etc/systemd/resolved.conf.d
cat >/etc/systemd/resolved.conf.d/sing-box-disable-stub.conf <<'EOF'
[Resolve]
DNSStubListener=no
EOF

systemctl restart systemd-resolved
sleep 2
ss -ltnup | grep -E ':53\b' || true

这里不要把 /etc/resolv.conf 改成公共 DNS，也不要乱删 resolv.conf。目的只是释放本机 53 端口，让 sing-box 监听 LAN DNS。

如果 53 端口被 dnsmasq、bind9、unbound、AdGuardHome 或其它服务占用，要先停掉或改它们的监听端口。sing-box 恢复后必须独占用于网关 DNS 的 53 端口。

还原文件

先给新机器现有文件做一个本地备份：

backup_stamp=$(date +%Y%m%d-%H%M%S)
mkdir -p /root/pre-sing-box-restore-$backup_stamp

[ -e /etc/sing-box ] && cp -a /etc/sing-box /root/pre-sing-box-restore-$backup_stamp/
[ -e /opt/singbox-rule-ui ] && cp -a /opt/singbox-rule-ui /root/pre-sing-box-restore-$backup_stamp/
[ -e /usr/local/bin/sing-box ] && cp -a /usr/local/bin/sing-box /root/pre-sing-box-restore-$backup_stamp/
for f in sing-box.service sing-box-tproxy.service singbox-rule-ui.service update-sing-box-rules-jsdelivr.service update-sing-box-rules-jsdelivr.timer; do
  [ -e /etc/systemd/system/$f ] && cp -a /etc/systemd/system/$f /root/pre-sing-box-restore-$backup_stamp/
done

然后按原路径复制。下面几行要在解压后的 sing-box-production-manual-restore-20260611 目录里执行：

cd /root/sing-box-production-manual-restore-20260611

cp -a files/etc/sing-box /etc/
cp -a files/opt/singbox-rule-ui /opt/
cp -a files/usr/local/bin/sing-box /usr/local/bin/
cp -a files/usr/local/bin/sing-box-gateway-info /usr/local/bin/

chown -R root:root /etc/sing-box /opt/singbox-rule-ui

修改新机器地址

备份包里的默认监听地址是脱敏恢复包的占位值，不一定等于新机器地址：

默认 LAN IPv4: 10.20.20.6
默认 LAN IPv6 DNS: fd00::6
DNS: 53
TProxy: 9888
Rule UI: 0.0.0.0:9091
Clash API: 10.20.20.6:9090

先找出新机器自己的地址。IPv4 通常看 ip route get 1.1.1.1 里的 src；IPv6 如果不需要 LAN IPv6 DNS，可以后面留空或删掉 dns-in-v6，小白环境建议先用机器已有的内网 IPv6：

ip -o -4 addr show scope global
ip -o -6 addr show scope global
ip route get 1.1.1.1

假设新机器 IPv4 是 10.20.20.16，要监听的 IPv6 DNS 地址是 fd88::1616，用下面命令替换配置里的监听地址。这里用 Python 做纯文本替换，避免 sed 遇到 IPv6 冒号、斜杠或不同发行版参数差异时出错：

old_v4='10.20.20.6'
new_v4='10.20.20.16'
old_v6='fd00::6'
new_v6='fd88::1616'

grep -R "$old_v4\|$old_v6" /etc/sing-box /usr/local/sbin -n || true

python3 - <<PY
from pathlib import Path
paths = [
    Path('/etc/sing-box/config.json'),
    Path('/etc/sing-box/manager/base.json'),
    Path('/usr/local/sbin/sing-box-tproxy-setup'),
    Path('/usr/local/sbin/refresh-sing-box-runtime-config'),
]
for path in paths:
    text = path.read_text(encoding='utf-8')
    text = text.replace('$old_v4', '$new_v4').replace('$old_v6', '$new_v6')
    path.write_text(text, encoding='utf-8')
PY

再检查网卡名。生产脚本里是 eth0：

ip -br link
grep -n '^IFACE=' /usr/local/sbin/sing-box-tproxy-setup

如果新机器网卡不是 eth0，例如是 ens18：

sed -i 's/^IFACE=.*/IFACE="ens18"/' /usr/local/sbin/sing-box-tproxy-setup

FakeIP 网段和 TProxy 规则不能随便改。公开恢复包默认 IPv4 FakeIP 是 28.0.0.0/8，IPv6 FakeIP 是 2001:2::/64；除非你完全理解 RouterOS、sing-box DNS fakeip 和 nftables 三边关系，否则不要只改一处。

检查配置

先让运行时配置按当前 UI 管理文件重新渲染：

/usr/local/sbin/refresh-sing-box-runtime-config

检查 sing-box 配置：

/usr/local/bin/sing-box check -c /etc/sing-box/config.json

如果这里失败，不要启动服务。常见原因是：

  • 新机器 IP 或 IPv6 地址没有改完整。
  • 网卡名不对。
  • 备份包是 amd64，机器却是 ARM。
  • 配置里的规则文件没复制完整。

启动 systemd 服务

加载并启用服务：

systemctl daemon-reload
systemctl enable sing-box-tproxy.service sing-box.service singbox-rule-ui.service update-sing-box-rules-jsdelivr.timer
systemctl restart sing-box-tproxy.service
systemctl restart sing-box.service
systemctl restart singbox-rule-ui.service
systemctl restart update-sing-box-rules-jsdelivr.timer

检查状态：

systemctl status --no-pager sing-box-tproxy.service
systemctl status --no-pager sing-box.service
systemctl status --no-pager singbox-rule-ui.service
systemctl status --no-pager update-sing-box-rules-jsdelivr.timer

检查端口：

ss -ltnup | grep -E ':(53|9090|9091|9888)\b' || true

正常情况下你应该看到：

  • sing-box 监听 LAN IPv4/IPv6 的 53。
  • sing-box 监听 Clash API 9090。
  • Rule UI 监听 9091。
  • TProxy 端口 9888 由 sing-box 使用。

验证 DNS 和 UI

从新机器本机验证 DNS。这里要查 sing-box 实际监听的 LAN IP，不要查 127.0.0.1，因为本项目默认不让 sing-box 监听 localhost：

new_v4='10.20.20.16'
dig @$new_v4 apple.com A
dig @$new_v4 example.com A

如果没有 dig，Ubuntu/Debian 可以装：

apt install -y dnsutils

也可以先用 sing-box-gateway-info 确认访问地址和密钥：

sing-box-gateway-info

从局域网电脑访问：

Rule UI:  http://新机器IP:9091/
Clash API: http://新机器IP:9090/
Clash UI:  http://新机器IP:9090/ui/
DNS: 新机器IP:53

Rule UI 登录 token 获取命令：

cat /etc/sing-box/rule-ui/token

如果你想换一个自己的 token：

install -d -m 755 /etc/sing-box/rule-ui
printf '%s\n' '换成你自己的-Rule-UI-token' > /etc/sing-box/rule-ui/token
chmod 600 /etc/sing-box/rule-ui/token
systemctl restart singbox-rule-ui.service
cat /etc/sing-box/rule-ui/token

打开 http://新机器IP:9091/ 后，把上面输出的 token 粘贴进去登录。生产使用前建议换成新的长随机 token。

Clash secret 获取命令：

python3 - <<'PY'
import json
cfg=json.load(open('/etc/sing-box/config.json'))
print(cfg.get('experimental', {}).get('clash_api', {}).get('secret', ''))
PY

如果要改 Clash secret，可以直接编辑 /etc/sing-box/manager/base.json 和 /etc/sing-box/config.json 里的 experimental.clash_api.secret，然后执行：

/usr/local/sbin/refresh-sing-box-runtime-config
/usr/local/bin/sing-box check -c /etc/sing-box/config.json
systemctl restart sing-box.service

路由器侧要改什么

新机器恢复成功后，路由器或客户端必须把 DNS 指向新机器。否则 sing-box 就算已经监听 53，客户端也不会用它。

如果你用 RouterOS 做主路由，还要确认：

  • DHCP 下发的 DNS 是新机器 IPv4。
  • IPv6 客户端如果要用网关 DNS，RA/DHCPv6 也要指向新机器 IPv6。
  • FakeIP IPv4 网段要路由到新机器。
  • FakeIP IPv6 网段要路由到新机器。
  • FastTrack 不要抢先放行本该进 TProxy 的连接。

这篇文章只讲 Linux 机器恢复。RouterOS 规则要按你自己的网段、接口和 FakeIP 池调整，不要直接复制旧生产环境的私有地址到另一个网络。

回滚

如果启动后不对，先停服务：

systemctl stop singbox-rule-ui.service
systemctl stop sing-box.service
systemctl stop sing-box-tproxy.service
nft delete table inet singbox_tproxy 2>/dev/null || true

恢复你开头保存的本地备份：

ls -ld /root/pre-sing-box-restore-*

找到对应目录后再手动复制回去。不要在没确认目录的情况下盲目删除 /etc/sing-box 或 /opt/singbox-rule-ui。

重新生成生产包的命令

下面这段是给自己重新生成私有生产包用的，不适合原样放到公开文章下载。生成后必须删除历史备份和运行缓存，并把真实节点、Rule UI token、Clash secret、生产 IPv6 FakeIP 前缀替换成模板值，才能公开分发。以后如果要重新从生产机生成同类包，可以在生产机 root 下执行：

stamp=$(date +%Y%m%d-%H%M%S)
work=/tmp/sing-box-production-manual-restore-$stamp
mkdir -p "$work/manifest" "$work/files/etc" "$work/files/opt" "$work/files/etc/systemd/system" "$work/files/usr/local/bin" "$work/files/usr/local/sbin"

{
  echo "backup_created=$stamp"
  echo "source_host=$(hostname)"
  echo "source_kernel=$(uname -a)"
  echo "sing_box_version_begin"
  /usr/local/bin/sing-box version || true
  echo "sing_box_version_end"
} > "$work/manifest/host-info.txt"

systemctl cat sing-box.service sing-box-tproxy.service singbox-rule-ui.service update-sing-box-rules-jsdelivr.service update-sing-box-rules-jsdelivr.timer > "$work/manifest/systemd-units-expanded.txt" 2>&1 || true
systemctl status --no-pager sing-box.service sing-box-tproxy.service singbox-rule-ui.service sing-box-tproxy.service update-sing-box-rules-jsdelivr.timer > "$work/manifest/systemd-status.txt" 2>&1 || true
ss -ltnup > "$work/manifest/listeners.txt" 2>&1 || true
ip rule show > "$work/manifest/ip-rule.txt" 2>&1 || true
ip route show table all > "$work/manifest/ip-route-all.txt" 2>&1 || true
nft list ruleset > "$work/manifest/nft-ruleset.txt" 2>&1 || true

cp -a /etc/sing-box "$work/files/etc/"
cp -a /opt/singbox-rule-ui "$work/files/opt/"
cp -a /usr/local/bin/sing-box "$work/files/usr/local/bin/"
for f in /usr/local/sbin/refresh-sing-box-runtime-config /usr/local/sbin/sing-box-tproxy-setup /usr/local/sbin/update-sing-box-rules-jsdelivr /usr/local/sbin/bootstrap-sing-box-config /usr/local/sbin/refresh-sing-box-tproxy-setup.disabled-20260606-radvd; do
  [ -e "$f" ] && cp -a "$f" "$work/files/usr/local/sbin/"
done
for f in /etc/systemd/system/sing-box.service /etc/systemd/system/sing-box-tproxy.service /etc/systemd/system/singbox-rule-ui.service /etc/systemd/system/update-sing-box-rules-jsdelivr.service /etc/systemd/system/update-sing-box-rules-jsdelivr.timer; do
  [ -e "$f" ] && cp -a "$f" "$work/files/etc/systemd/system/"
done

find "$work/files" -printf "%M %u %g %s %TY-%Tm-%Td %TH:%TM %p\n" | sed "s#$work/##" | sort > "$work/manifest/file-list.txt"
cat > "$work/README-BACKUP.txt" <<'EOF'
这是生产 sing-box 网关的纯手动恢复备份包。
恢复时不要依赖 GitHub 仓库；按文章流程把 files/ 下的路径复制回新机器。
敏感信息包含节点、Rule UI token、Clash secret、规则和生产 UI，请只放在可信机器。
EOF

cd /tmp
tar -czpf "$work.tar.gz" "$(basename "$work")"
sha256sum "$(basename "$work.tar.gz")" > "$work.tar.gz.sha256"
ls -lh "$work.tar.gz" "$work.tar.gz.sha256"
cat "$work.tar.gz.sha256"

把 .tar.gz 和 .sha256 一起复制走，恢复前永远先校验。

最后检查清单

恢复完成后，至少确认这些项：

  • sha256sum -c 通过。
  • sing-box check -c /etc/sing-box/config.json 通过。
  • systemctl status 里 sing-box、sing-box-tproxy、singbox-rule-ui 都正常。
  • ss -ltnup 里 53 端口由 sing-box 占用。
  • Rule UI 能打开，token 能登录。
  • Clash API/Clash UI 能打开。
  • 局域网客户端 DNS 已指向新机器。
  • RouterOS 或主路由已把 FakeIP 网段送到新机器。

这套流程的核心不是“复制文件就完事”，而是恢复运行边界：文件、二进制、systemd、53 端口、TProxy、路由器 DNS/路由必须一起对齐。
