Syncthing 实现 Ubuntu 到 Ubuntu/Alpine 的单向安全同步

在 Ubuntu 源端和 Ubuntu 或 Alpine 备份端之间部署 Syncthing，将 /root/data/backup 目录配置为仅发送到仅接收的单向加密同步。

这篇记录整理一个比较实用的备份场景：A 机器是 Ubuntu 或 Debian，作为源端；B 机器可以是 Ubuntu、Debian 或 Alpine，作为备份端；目标是把 A 机器上的 /root/data/backup 单向同步到 B 机器，不让备份端的改动反向影响源端。

Syncthing 是一个持续文件同步工具，可以在两台或多台机器之间实时同步文件。它没有中心服务器，数据只保存在自己的设备上；设备之间的通信使用 TLS 加密，每台设备也通过独立证书身份识别。这个特性很适合自建备份链路，但 Web 管理面板仍然要谨慎暴露，公网环境下更推荐通过 SSH 隧道访问。

适用场景

  • 源端目录：Ubuntu/Debian 上的 /root/data/backup
  • 备份端目录：Ubuntu/Debian 或 Alpine 上的 /root/data/backup
  • 同步模式：源端 仅发送，接收端 仅接收
  • 运行用户：本文按 root 管理目录来写
  • 管理方式：优先通过 SSH 隧道访问 Syncthing Web GUI

如果只是每天固定时间做一次备份，rsync + SSH 也很适合；如果希望目录变动后实时同步，并且希望 Syncthing 负责设备发现、传输加密和状态管理，可以使用本文方案。

一、Ubuntu 源端配置

1. 安装 Syncthing

sudo apt update
sudo apt install syncthing -y

为 root 生成配置文件：

syncthing -generate="/root/.config/syncthing"

2. 启动 root 级 systemd 服务

Ubuntu 的包通常会提供 syncthing@.service 模板，可以直接以指定用户启动：

systemctl daemon-reload
systemctl enable --now syncthing@root.service
systemctl status syncthing@root.service

确认服务运行后，查看监听端口：

ss -tunlp | grep syncthing

Syncthing 的 Web GUI 默认监听 127.0.0.1:8384，同步通信端口通常是 22000/tcp 和 22000/udp。如果日志中显示了自定义监听端口，以日志为准。

3. 推荐：保持 Web GUI 只监听本机

不要把管理面板直接暴露到公网。保持配置中的 GUI 地址为本地监听：

grep -n "<address>" /root/.config/syncthing/config.xml

如果已经改成了 0.0.0.0:8384，可以恢复为：

sed -i 's/<address>0.0.0.0:8384<\/address>/<address>127.0.0.1:8384<\/address>/g' /root/.config/syncthing/config.xml
systemctl restart syncthing@root.service

然后从自己的电脑建立 SSH 隧道访问管理面板：

ssh -N -L 8384:127.0.0.1:8384 root@你的Ubuntu服务器IP

隧道保持运行后，在本地浏览器打开：

http://127.0.0.1:8384

这样服务器的 8384 不需要对公网开放，管理流量会包在 SSH 加密连接里。

4. 只放行同步通信端口

如果 Ubuntu 使用 UFW，只需要放行 Syncthing 的同步通信端口：

ufw allow 22000/tcp
ufw allow 22000/udp
ufw reload

如果你的 Syncthing 日志显示使用了其他端口，例如 29029，就按实际端口放行：

ufw allow 29029/tcp
ufw allow 29029/udp
ufw reload

二、Ubuntu/Debian 备份端配置

如果接收端也是 Ubuntu 或 Debian，配置方式和源端基本一致，只是 Web 面板里要把共享目录设为 仅接收。

1. 安装 Syncthing

sudo apt update
sudo apt install syncthing -y

为 root 生成配置文件：

syncthing -generate="/root/.config/syncthing"

2. 启动 root 级 systemd 服务

systemctl daemon-reload
systemctl enable --now syncthing@root.service
systemctl status syncthing@root.service

查看监听端口：

ss -tunlp | grep syncthing

正常情况下，你会看到类似下面的监听：

tcp   LISTEN 0 4096 127.0.0.1:8384 0.0.0.0:* users:(("syncthing",pid=1234,fd=14))
tcp   LISTEN 0 4096 *:22000          *:*       users:(("syncthing",pid=1234,fd=11))
udp   UNCONN 0 0    *:22000          *:*       users:(("syncthing",pid=1234,fd=13))
udp   UNCONN 0 0    0.0.0.0:21027    0.0.0.0:* users:(("syncthing",pid=1234,fd=19))

其中：

  • 127.0.0.1:8384 是 Web 管理面板，只监听本机是推荐状态。
  • 22000/tcp 和 22000/udp 是设备之间的数据同步端口，需要放行。
  • 21027/udp 主要用于局域网发现，公网 VPS 场景不是重点。

3. 通过 SSH 隧道访问接收端面板

接收端也建议保持 Web GUI 只监听 127.0.0.1:8384。从本地电脑建立隧道时，为了避免和源端的本地 8384 冲突，可以把接收端映射到本地 8385：

ssh -N -L 8385:127.0.0.1:8384 root@你的Ubuntu接收端IP

然后在本地浏览器打开：

http://127.0.0.1:8385

4. 只放行同步通信端口

如果接收端使用 UFW，放行 Syncthing 同步端口：

ufw allow 22000/tcp
ufw allow 22000/udp
ufw reload

如果云服务器还有安全组，也要在安全组里放行 22000/tcp 和 22000/udp。Web 管理端口 8384 不建议放行到公网。

三、Alpine 备份端配置

1. 安装 Syncthing

apk update
apk add syncthing

生成 root 配置：

syncthing -generate="/root/.config/syncthing"

2. 配置 OpenRC 服务

Alpine 使用 OpenRC。为了让 Syncthing 使用 root 的配置目录运行，可以重写 /etc/init.d/syncthing：

cat << 'EOF' > /etc/init.d/syncthing
#!/sbin/openrc-run

name="syncthing"
command="/usr/bin/syncthing"
command_user="root:root"
command_args="-home=/root/.config/syncthing -gui-address=127.0.0.1:8384 -no-browser -no-restart"
command_background="yes"
pidfile="/run/syncthing.pid"

depend() {
    need net
}
EOF

赋权并启动：

chmod +x /etc/init.d/syncthing
rc-service syncthing stop
killall -9 syncthing
rc-update add syncthing default
rc-service syncthing start
rc-service syncthing status

同样建议 Alpine 的 Web GUI 只监听 127.0.0.1:8384。需要访问时，在本地电脑建立另一条隧道：

ssh -N -L 8385:127.0.0.1:8384 root@你的Alpine服务器IP

然后浏览器打开：

http://127.0.0.1:8385

3. Alpine 防火墙放行

如果 Alpine 侧启用了防火墙，只放行 Syncthing 同步通信端口即可。以 awall 或其他防火墙为准，端口仍然优先看 Syncthing 日志。常见默认端口是：

22000/tcp
22000/udp

四、建立设备信任

分别打开两台机器的 Syncthing Web GUI。

在 B 机器，也就是 Ubuntu/Debian 或 Alpine 备份端：

  1. 点击 操作。
  2. 点击 显示 ID。
  3. 复制设备 ID。

回到 A 机器，也就是 Ubuntu 源端：

  1. 点击 添加远程设备。
  2. 粘贴备份端的设备 ID。
  3. 保存。

然后回到备份端面板，等待连接请求弹出，点击 添加设备 接受 Ubuntu 源端。

如果两台机器都在公网 VPS 上，通常会自动发现并连接。如果连接不上，优先检查：

  • 两端同步通信端口是否放行。
  • Syncthing 日志中实际监听的端口是多少。
  • 云厂商安全组是否也放行了对应端口。
  • 两端时间是否准确，证书认证对系统时间比较敏感。

五、配置单向同步目录

1. Ubuntu 源端设置为仅发送

在 Ubuntu 的 Syncthing 面板中：

  1. 点击 添加文件夹。
  2. 文件夹路径 填写 /root/data/backup。
  3. 在 共享 选项卡中勾选备份端。
  4. 在 高级 选项卡中，将 文件夹类型 设置为 仅发送 (Send Only)。
  5. 保存。

仅发送 的意思是：Ubuntu 是权威源端。源端新增、修改、删除会同步到对端，但对端的本地改动不会被当成正常变更反向覆盖源端。

2. Ubuntu/Debian 或 Alpine 备份端设置为仅接收

在备份端的 Syncthing 面板中：

  1. 等待 Ubuntu 发来的共享目录提示。
  2. 点击接受。
  3. 本地路径填写 /root/data/backup。
  4. 在 高级 选项卡中，将 文件夹类型 设置为 仅接收 (Receive Only)。
  5. 保存。

仅接收 的意思是：备份端只接收源端下发的数据。如果有人在备份目录里手动改文件，Syncthing 会把它标记为本地变更，而不是推回源端。

六、验证同步

在 Ubuntu 源端创建测试文件：

mkdir -p /root/data/backup
date > /root/data/backup/syncthing-test.txt

观察两端 Web GUI，等待状态变成 最新。然后在备份端检查：

cat /root/data/backup/syncthing-test.txt

再测试删除同步：

rm /root/data/backup/syncthing-test.txt

如果源端删除文件，备份端也会跟着删除。这个行为适合“镜像式备份”，但不适合“防误删归档”。如果担心源端误删同步到备份端，应额外在备份端配置快照、版本控制或定时归档。

1. 对比两端目录大小和文件数

当 Syncthing 面板显示远程设备和文件夹都是 最新，但当前速率是 0 B/s 时，不一定是没有传输成功，也可能是已经同步完成、没有新数据需要传。

可以分别在源端和备份端执行：

du -sh /root/data/backup
find /root/data/backup -type f | wc -l

例如源端输出：

5.0G    /root/data/backup
4

备份端输出：

5.6G    /root/data/backup
5

这类结果不一定表示同步失败。du -sh 看的是磁盘实际占用，不同文件系统的块大小、稀疏文件、压缩、元数据或临时文件都可能让显示值略有差异。Syncthing 面板里常见的 GiB 和系统 du 里的 G 也不是完全同一种显示口径。

如果文件数有差异，再进一步对比文件相对路径和字节数：

find /root/data/backup -type f -printf '%P %s\n' | sort

两端都执行一次。如果源端的业务文件在备份端都存在，大小也一致，Syncthing 面板又显示 最新，就可以认为同步已经完成。备份端多出来的文件如果是 Syncthing 标记文件、历史残留文件或本地临时文件，需要按实际情况判断是否清理。

2. 测试后续增量同步

想确认后续实时同步是否正常，可以在源端新增一个很小的测试文件：

date > /root/data/backup/test-sync.txt

然后在备份端检查：

ls -lh /root/data/backup/test-sync.txt
cat /root/data/backup/test-sync.txt

如果备份端能看到这个文件，说明增量同步正常。测试完成后，在源端删除它：

rm /root/data/backup/test-sync.txt

再确认备份端也同步删除即可。

七、内网备份端的 RouterOS 端口映射与直连排查

如果备份端在家里或办公室内网，例如备份端 IP 是 10.20.20.8，而源端是公网 VPS，那么只在备份端系统里放行 22000 还不够。外网源端想直连内网备份端，前端路由器也要把 Syncthing 同步端口 NAT 到备份端。

下面以 RouterOS 为例：

  • WAN 接口：pppoe-out1
  • 备份端内网 IP：10.20.20.8
  • Syncthing 同步端口：22000/tcp 和 22000/udp

1. RouterOS 添加 dstnat 端口映射

/ip firewall nat
add chain=dstnat in-interface=pppoe-out1 protocol=tcp dst-port=22000 action=dst-nat to-addresses=10.20.20.8 to-ports=22000 comment="Syncthing TCP to 10.20.20.8"
add chain=dstnat in-interface=pppoe-out1 protocol=udp dst-port=22000 action=dst-nat to-addresses=10.20.20.8 to-ports=22000 comment="Syncthing UDP to 10.20.20.8"

如果你的 RouterOS 使用 dst-address-type=local 统一匹配本机公网地址，也可以按现有规则风格写成：

/ip firewall nat
add chain=dstnat dst-address-type=local protocol=tcp dst-port=22000 action=dst-nat to-addresses=10.20.20.8 to-ports=22000 comment="Syncthing TCP to 10.20.20.8"
add chain=dstnat dst-address-type=local protocol=udp dst-port=22000 action=dst-nat to-addresses=10.20.20.8 to-ports=22000 comment="Syncthing UDP to 10.20.20.8"

两种写法选一种即可，不要重复建太多同类规则。

2. RouterOS 放行 forward

如果防火墙 forward 链最后有 drop all other forward，就需要确保端口转发流量在 drop 前被放行。

可以单独放行 Syncthing：

/ip firewall filter
add chain=forward in-interface=pppoe-out1 protocol=tcp dst-address=10.20.20.8 dst-port=22000 action=accept comment="Allow Syncthing TCP to 10.20.20.8"
add chain=forward in-interface=pppoe-out1 protocol=udp dst-address=10.20.20.8 dst-port=22000 action=accept comment="Allow Syncthing UDP to 10.20.20.8"

如果已有类似下面这种通用规则，并且 pppoe-out1 已经在 WAN interface list 里，那么单独的 Syncthing filter 规则不是必须的：

/ip firewall filter
add chain=forward action=accept connection-nat-state=dstnat in-interface-list=WAN comment="Allow DSTNAT Port Forwarding"

规则顺序很重要：放行规则必须在最后的 drop 规则前面。

3. 检查 RouterOS 规则

打印 NAT 和 filter 规则：

/ip firewall nat print detail without-paging
/ip firewall filter print detail without-paging

检查 Syncthing 规则是否有命中计数：

/ip firewall nat print stats where comment~"Syncthing"
/ip firewall filter print stats where comment~"Syncthing"

如果外部源端正在尝试连接，packets 和 bytes 应该会增长。只要 NAT 和 filter 计数在增长，说明 RouterOS 端口转发基本已经生效。

还要确认 WAN 口拿到的是公网 IPv4：

/ip address print where interface=pppoe-out1

如果这里显示的是 10.x.x.x、100.64.x.x、172.16.x.x 到 172.31.x.x、192.168.x.x 这类地址，说明上游可能是内网或 CGNAT。此时 RouterOS 本机端口映射无法从公网直接生效，需要改用公网 IPv6、WireGuard、Tailscale、ZeroTier，或者继续使用 Syncthing relay。

4. Syncthing 里指定直连地址

如果备份端所在网络有公网 IPv4，并且 RouterOS 已经映射到 10.20.20.8:22000，建议在源端 Syncthing 里编辑备份端设备：

高级 -> 地址

把 dynamic 改成：

tcp://你的公网IP:22000

保存后暂停/恢复该远程设备，或者重启源端 Syncthing：

systemctl restart syncthing@root.service

回到 Syncthing 主界面，点开远程设备详情，正常直连时连接类型应从：

中继广域网

变成类似：

TCP WAN

如果仍然显示 中继广域网，说明它还在走 relay。relay 能用，但速度通常很慢，初次同步几个 GiB 时会非常明显。

5. 在备份端确认是否已经直连

在备份端 10.20.20.8 上查看 Syncthing 连接：

ss -tunp | grep syncthing

如果看到类似下面这类连接，说明外部源端已经通过 RouterOS 打到了备份端的 22000：

tcp ESTAB 0 0 10.20.20.8:22000 142.171.220.24:29029 users:(("syncthing",pid=2635,fd=22))

这里的重点是本地端口为 10.20.20.8:22000，对端是源端公网 IP。对端端口不一定是 22000，这很正常。

如果看到的是类似下面这种：

tcp ESTAB 0 0 10.20.20.8:50454 101.132.255.222:22067 users:(("syncthing",pid=2635,fd=11))

这通常是 relay 中继连接，不代表直连已经成功。

6. 判断是不是真的在传数据

Syncthing 面板里的 下载速率 和 上传速率 要结合当前设备角色看。源端向备份端发送数据时，源端主要看 上传速率；备份端主要看 下载速率。

如果面板显示 同步中 (0%, 4.98 GiB)，但速率长期是 0 B/s，先点开远程设备看连接类型。如果显示 中继广域网，优先解决直连；如果已经是 TCP WAN，再检查文件夹状态、权限和扫描进度。

在备份端可以用 iftop 看实时流量。Alpine 安装：

apk add iftop

Ubuntu/Debian 安装：

apt install iftop -y

先找默认出口网卡：

ip route get 1.1.1.1

然后只看 Syncthing 同步端口：

iftop -i 你的网卡名 -f "port 22000"

如果 iftop 里主要看到的是本地浏览器访问 10.20.20.8:8384，那只是 Web GUI 流量，不是同步数据。真正的同步流量应该能看到备份端与源端公网 IP 之间的连接。

八、安全建议

  • 不要把 8384 Web GUI 管理端口直接暴露到公网。
  • 如果临时开放 Web GUI，至少启用 GUI 用户名和强密码，并优先启用 HTTPS。
  • 生产环境建议通过 SSH 隧道访问管理面板。
  • 同步端口只开放给必要来源更稳妥，云安全组和系统防火墙都要检查。
  • 源端使用 仅发送，备份端使用 仅接收，不要配置成默认的双向同步。
  • 对重要备份目录，建议在备份端再做快照或版本保留，避免误删被实时同步。

九、可选：什么时候用 rsync + SSH

如果你不需要实时同步，只需要每天凌晨把源端目录复制到备份端，rsync + SSH 更简单：

ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519_backup -N ""
ssh-copy-id -i ~/.ssh/id_ed25519_backup.pub root@备份端IP

测试同步：

rsync -avz --delete -e "ssh -i ~/.ssh/id_ed25519_backup" /root/data/backup/ root@备份端IP:/root/data/backup/

加入定时任务：

crontab -e

例如每天凌晨 2 点执行：

0 2 * * * /usr/bin/rsync -az --delete -e "ssh -i /root/.ssh/id_ed25519_backup" /root/data/backup/ root@备份端IP:/root/data/backup/ >> /root/backup_sync.log 2>&1

这里也要注意源目录末尾的 /：/root/data/backup/ 表示同步目录里的内容，而不是把整个 backup 目录再套一层传过去。
